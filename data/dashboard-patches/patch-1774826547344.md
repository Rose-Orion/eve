## Diagnosis

The `attention-cta` "Review →" buttons in the Tower's "Needs Attention" section are built correctly for gate approvals — they call `router.go('gate1', {id: floorId})`. However, the action trail shows the router navigated to `gate1?id=b090c7ea-ef9c-4bb3-abdf-62897e1224e8`, which **is** `gate1` with the floor ID. The bug is that `viewGate1` renders a **"Pick This"** button that calls `router.go('gate1-voice', {...})`, but the **gate1 view itself** also has a sub-navigation flow (gate1 → gate1-voice → gate1-strategy → approveFoundation). The owner is reporting the destination is wrong — they expect clicking "Review →" from the Tower to take them to the full gate1 review flow, which it does, but the `gate1` route in the `render()` map is missing the `gate1-voice` and `gate1-strategy` sub-routes only if the router somehow loops back.

After careful re-reading: the real issue is that **the `attention-cta` button in the Tower's approval items uses `reviewDest`**, which for a `gate` approval with `phaseNumber === 3` resolves to `router.go('gate1', {id:'${floor.id}'})`. The owner sees `gate1?id=b090c7ea...` in the URL-equivalent — that **is** gate1. But the owner reports it's "the wrong destination." 

Looking at the action trail timestamp and the approval routing logic more carefully: for **non-gate approvals** (`a.type !== 'gate'`), `reviewDest` is set to `router.go('floor', {id:'${a.floorId}'})`. The bug is that **when `a.type === 'gate'` but `a.phaseNumber` doesn't match 3, 6, or 8**, the fallback is `'gate1'` — but the approval in question may be a task-output approval that should route to a task output viewer, not gate1 at all. More critically: the fallback `else` branch for unknown gate phases uses `'gate1'` as default even for approvals that have nothing to do with Foundation review.

The most precise diagnosis: **the default/fallback `gateView` in the Tower's `approvalItems` map is `'gate1'`**, which means any approval with an unexpected or missing `phaseNumber` silently routes to `gate1` instead of the appropriate review screen. Since the owner reports arriving at `gate1?id=b090c7ea-ef9c-4bb3-abdf-62897e1224e8` and that being wrong, the approval triggering this CTA likely has a `phaseNumber` that doesn't match 3, 6, or 8 — and the fallback incorrectly sends them to `gate1`. The correct destination for an unrecognized gate phase should be the floor's detail page, or the approval should be logged and the user routed to `floor` view where they can see pending items.

## Patch

### Change 1: Fix the Tower approval CTA fallback — unknown gate phases route to floor detail, not gate1

**What it changes:** The `gateView` fallback in `viewTower()`'s `approvalItems` map currently defaults to `'gate1'` for any unrecognized `phaseNumber`. This causes any non-Foundation gate approval to incorrectly loop to the Foundation review screen. The fix routes unknown gate approvals to the floor detail page instead, where all pending actions are visible.

**Why:** Gate phase numbers 3 = Foundation, 6 = Website/QA, 8 = Ads. Any other value is unknown and should not silently drop the owner on gate1. The floor detail page shows all pending gates prominently and lets the owner navigate from there.

File: `public/app.js`

Find:
```javascript
    let reviewDest;
    if (a.type === 'gate') {
      const gateView = a.phaseNumber === 3 ? 'gate1'
        : a.phaseNumber === 6 ? 'gate2'
        : a.phaseNumber === 8 ? 'gate3'
        : 'gate1'; // Unknown gate phase — default to Foundation review
      const floor = state.floors.find(f => f.id === a.floorId);
      reviewDest = floor
        ? (gateView ? `router.go('${gateView}', {id:'${floor.id}'})` : `router.go('floor', {id:'${floor.id}'})`)
        : `router.go('tower')`;
    } else {
      // Non-gate approvals (task reviews, feedback) — route to floor detail
      reviewDest = `router.go('floor', {id:'${a.floorId}'})`;
    }
```

Replace with:
