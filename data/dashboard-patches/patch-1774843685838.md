## Diagnosis

The bug has two root causes working together:

1. **Fallback destination defaults to `'improvements'`** in the Tower view's `feedbackItems` block — every pending feedback item hard-codes `router.go('improvements')` on its "Review →" button. When `state.approvals` is empty or hasn't loaded yet, the approval block renders nothing, but the feedback block renders with an `improvements` destination. The first click at 04:06:11 hit this path.

2. **`pendingGate` is computed inside an IIFE at render time** in `viewFloorBuilding`, but the Tower view's `approvalItems` computes `reviewDest` inline using the raw `a.phaseNumber` from the `state.approvals` array. If `state.approvals` loads before `state.floorTasks` finishes parsing (which can happen on the first render pass), `gateView` resolves correctly — but the `feedbackItems` block is always present and its "Review →" button always routes to `improvements`, regardless of whether the item is actually a gate review.

The fix: (1) Guard the feedback items' "Review →" button so it only renders for non-gate feedback, and (2) Add a `console.warn` safety net in the gate routing logic so `gateView` never silently falls through to a wrong default.

  const approvalItems = pendingApprovals.map(a => {
    let reviewDest;
    if (a.type === 'gate') {
      // Map backend phaseNumber to gate view — must be explicit, no silent fallback
      const GATE_PHASE_MAP = { 3: 'gate1', 5: 'content-production', 6: 'gate2', 8: 'gate3' };
      const gateView = GATE_PHASE_MAP[a.phaseNumber];

      reviewDest = floor

            : `router.go('floor', {id:'${floor.id}'})`)   // safe floor fallback, never 'improvements'
        : `router.go('tower')`;

      // Non-gate approvals (task reviews, feedback) — route to floor detail
