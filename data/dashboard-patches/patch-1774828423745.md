## Diagnosis

The bug is in `viewFloorBuilding()`. When a content item has status `'production ready to view'`, the floor page renders a "Review →" button that calls `router.go('gate1', {id:'${floor.id}'})` — but this is reusing the gate review banner logic, which routes all pending reviews through the gate routing system (gate1/gate2/gate3). For the specific floor ID `b090c7ea-ef9c-4bb3-abdf-62897e1224e8`, the content item with status `'production ready to view'` is being incorrectly treated as a gate review item.

Looking at the code more carefully: the floor-level "Review →" banner and button are driven entirely by `pendingGate` (from `state.approvals`) and the gate routing block. There is **no separate handling** for content items with status `'production ready to view'` — they fall through to the gate routing path, causing them to route to `gate1` instead of a dedicated content view route.

The fix needs to: (1) detect when a floor task/content item has status `'production ready to view'`, and (2) route its "Review →" button to a `'content-review'` route (passing the floor ID and item ID), rather than `'gate1'`.

Since the codebase has no `content-review` view yet, and the owner's intent is to view production-ready content (not a gate), the correct minimal fix is to:
- Detect `'production ready to view'` tasks in `state.floorTasks[floor.id]`
- Render a separate, correctly-routed "Review →" button for those items
- Ensure the existing gate banner is NOT triggered for these content items
- Add a stub `viewContentReview` to the router so navigation doesn't break
