## Diagnosis

The "Review Content Production" button is broken because `content-production` is mapped to `viewGate1Strategy` in the router (a placeholder), and the approval routing in `viewFloorBuilding` uses a `GATE_PHASE_MAP` where **phase 5 maps to `'content-production'`** — but when `router.go('content-production', {id})` fires, `viewGate1Strategy` renders with `router.params.id` which is correct, however the **event listener binding is the core issue**: the `Review →` button in the attention banner and the floor's btn-row both use inline `onclick` strings that are generated at render time with the floor ID baked in. The actual problem is that `content-production` has **no dedicated view** — it silently falls through to `viewGate1Strategy` which is Gate 1 Step 3 (foundation approval), not a content review flow. When a phase-5 approval exists, clicking "Review →" navigates to `gate1-strategy` which calls `approveFoundation()` — a completely wrong action that would re-approve gate 3 instead of gate 5.

The fix requires: (1) a dedicated `viewContentProduction` function that handles phase-5 content review, (2) correct routing in `render()`'s views map, and (3) an `approveContentProduction` function that calls `/api/floors/${id}/approve-gate/5`.

async function approveContentProduction(id) {
  toast('Approving content production…', 'info');
  const result = await api('POST', `/api/floors/${id}/approve-gate/5`, {});
  if (result?.success || result) {
    const nextPhase = result?.nextPhase || 6;
    setLocalPhase(id, nextPhase);
    const floorIdx = state.floors.findIndex(f => f.id === id);
    if (floorIdx !== -1) {
      state.floors[floorIdx] = { ...state.floors[floorIdx], currentPhase: nextPhase };
    }
    await loadState();
    toast('Content approved! Moving to Staging & QA.', 'success');
    router.go('floor', { id });
  } else {
    toast('Approval failed — try again.', 'error');
  }
}
