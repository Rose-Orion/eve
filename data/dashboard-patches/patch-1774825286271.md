## Diagnosis

The "Review →" button on the floor detail page is inside a `div` that has an `onclick` routing to the correct gate view, but the **inner `<button>`** itself has no `onclick` handler — so click events on the button text bubble up and are caught by the parent `div`'s `onclick`, which *does* route correctly. However, looking more carefully at the bottom `btn-row` section's Review button (line near the end of `viewFloorBuilding`), it constructs the gate route string inline but has a fallback of `'floor'` as the view name — meaning when `pendingGate` is undefined or has an unrecognized `phaseNumber`, it routes back to `'floor'` with the same `id`, re-rendering the current floor view.

The specific floor `b090c7ea-ef9c-4bb3-abdf-62897e1224e8` has a `pendingGate` whose `phaseNumber` doesn't match 3, 6, or 8 (the three mapped gates), so the ternary chain falls through to `'floor'` — sending the user right back to the floor detail page instead of a review screen.

The fix is to make the fallback route to `'gate1'` (the most appropriate default for an unrecognized pending gate during early-phase building) rather than `'floor'`.
