## Diagnosis

The `Review →` button in `viewFloorBuilding` routes to `gate1` unconditionally — it hardcodes `'gate1'` in the bottom `btn-row` section regardless of what `pendingGate.phaseNumber` actually is. The floor `b090c7ea-ef9c-4bb3-abdf-62897e1224e8` has content marked as production-ready, which corresponds to phase 8 (the Content/Staging gate → `gate3`), but the button always sends the owner to `gate1` (Foundation Review) instead.

Looking at the code, there are **two** places in `viewFloorBuilding` where the bottom Review button is rendered. The inline banner (pulse card) correctly computes `gateView` using `pendingGate?.phaseNumber`, but the `btn-row` at the bottom has a hard-wired `gate1` fallback that bypasses the phase lookup entirely — it re-runs the ternary but with a copy-paste error that always resolves to `'gate1'`.

Wait — those are identical. Let me trace the actual rendered string more carefully. The problem is that `gateView` is a **JavaScript variable** computed in the block above, but the `btn-row` is in a **separate template literal scope** and re-inlines the ternary from scratch. The inline banner block declares `const gateView = ...` inside its own IIFE, so it's not accessible to the `btn-row`. The `btn-row` re-derives it inline — and looking closely at it, it reads:

```
pendingGate?.phaseNumber === 3 ? 'gate1' : pendingGate?.phaseNumber === 6 ? 'gate2' : pendingGate?.phaseNumber === 8 ? 'gate3' : 'gate1'
```

That logic is actually correct for known phase numbers. The real risk is: if the content-production-ready floor has `pendingGate.phaseNumber` that is **not** 3, 6, or 8 — e.g. it's `5` or `9` — the fallback is `gate1`. The fix is to extract `gateView` to a variable that's available to the whole `viewFloorBuilding` function scope, and use it consistently everywhere.
