## Diagnosis

**Issue 1 — Approval notifications not surfacing:** The polling loop in `init()` only triggers `render()` when `router.current === 'tower' || router.current === 'floor'`. If the owner is on any other view when a new approval arrives, the notification badge and alert banner never update. More critically, the poll hash comparison includes `state.approvals` but the re-render guard excludes views like `notifications`, `gate1`, etc. — so even on tower, if the owner doesn't navigate away and back, a freshly-arrived approval might not trigger a re-render because the hash *did* change but the view isn't in the allowed list.

**Issue 2 — Budget display not updating reactively:** The `refreshFloorBudget()` helper is only triggered when a task transitions to `completed` *and* its `taskType` is in the `BUDGET_TASK_TYPES` set (`budget-plan`, `budget-plan-redo`, `financial-projection`). Spend events from the backend update `spentCents` directly on the floor record — they are not tied to task completions. The polling loop re-fetches `/api/floors` but then the `loadState()` `spentCents` normalisation is correct; however the Supabase realtime handler maps `spent_cents` → `spentCents` correctly. The real bug: the polling render guard (`if (router.current === 'tower' || router.current === 'floor')`) means that after `loadState()` updates `state.floors[i].spentCents`, the floor view only re-renders if you're already on those exact views. If you're on `floor` but the hash didn't change on the previous tick (because spend happened between two polls), the budget widget stays stale. Additionally, the Supabase realtime `floors` subscription does re-render on `['tower', 'floor']` — but only if the `updated` row's `spent_cents` field is included in the postgres change payload (which requires `REPLICA IDENTITY FULL` on the table, something not guaranteed). The safest fix is: (a) always re-render the current view on any `state.floors` change, and (b) add `spent_cents`/`spentCents` to the poll hash so any spend event forces a re-render.

### Change 5: Notify owner when new approvals arrive mid-session

**What:** The approval badge and alert banner only appear on the next `render()`. But if the owner is actively watching the tower and a new approval arrives during a poll cycle, the current guard means render fires — good. However there's no *active notification* (e.g. toast) to snap the owner's attention. We add a lightweight approval-arrival detector that fires a toast + sound cue when the approval count increases, ensuring approvals never sit silently even if the owner is on a non-tower view.

File: `public/app.js`

Find:
```javascript
// Snapshot of spentCents per floor from the previous poll — used to detect spend events

```

Replace with:
```javascript
// Snapshot of spentCents per floor from the previous poll — used to detect spend events
const _prevSpentCents = {};

// Snapshot of pending approval IDs from the previous poll — used to surface new approval toasts
const _prevApprovalIds = new Set();

// Called after every loadState() to detect newly-arrived approvals and
// surface them as an owner-facing toast notification.
function checkForNewApprovals() {
  const pending = state.approvals.filter(a => a.status === 'pending');
  const newArrivals = pending.filter(a => !_prevApprovalIds.has(a.id));

  if (newArrivals.length > 0 && _prevApprovalIds.size > 0) {
    // Only toast on arrivals after the first load (size > 0 guard prevents
    // a flood of toasts on initial page load when all approvals are "new").
    for (const a of newArrivals) {
      const floorName = floorDisplayName(state.floors.find(f => f.id === a.floorId)) || 'your floor';
      const label = a.phaseName || a.title || 'Review needed';
      toast(`🔴 ${label} — ${floorName} is waiting for your approval`, 'info');
    }
  }

  // Rebuild snapshot from current pending set
  _prevApprovalIds.clear();
  for (const a of pending) _prevApprovalIds.add(a.id);
}
```
