## Diagnosis

The bug is in `viewFloorBuilding()` — specifically the **"Review →" button and the pulsing banner** inside the gate-ready block. When a pending gate approval exists for floor `b090c7ea-ef9c-4bb3-abdf-62897e1224e8`, the `pendingGate?.phaseNumber` check correctly maps phase 3 → `'gate1'`. However, the **bottom `btn-row`** section re-runs the same ternary inline and produces a string like `router.go('gate1', ...)` — but there is a subtle secondary issue: the **pulsing banner's `onclick`** attribute on the wrapping `<div>` sets the route correctly, but the **`<button>` inside it** (`Review →`) has no `onclick` of its own and inherits the div click, which is correct.

The actual root cause is in the **`btn-row` at the bottom** of `viewFloorBuilding`. The gate view name is re-computed inline with a long ternary that is duplicated verbatim — and because JavaScript template literals evaluate eagerly, if `pendingGate` is `undefined` at the moment the string is built (e.g., a race between `loadState()` and `render()` on the specific floor), the ternary falls through to `'gate1'` correctly — **but the `btn-row` button is actually routing to `'gate1'` with no `id` param when `pendingGate` is falsy**, stripping the floor ID. Let me trace more carefully…

Re-reading: the `btn-row` button at the bottom of `viewFloorBuilding` produces:

```js
router.go('gate1', {id:'${floor.id}'})
```

This is hardcoded to `'gate1'` regardless of `pendingGate.phaseNumber`. For floor `b090c7ea-ef9c-4bb3-abdf-62897e1224e8`, if the pending approval's `phaseNumber` is NOT 3 (e.g., it is phase 5 or 6 pointing to content production), this button **always** routes to `gate1` (Foundation/Brand page) instead of the correct gate. The banner's `<div onclick>` does the same hardcoded `gate1` route.

The fix: extract the correct `gateView` variable once and reuse it throughout the banner, button, and `btn-row`.
