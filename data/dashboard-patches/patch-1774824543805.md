## Diagnosis

In `viewFloorBuilding`, the LevelUp Quest (Phase 4) tasks include a Content Production task. When the owner clicks its 'Review' button, it routes to `gate1` (brand info) instead of a content production review screen. The root cause is in the `viewTower` approval routing logic: any gate approval with `phaseNumber === 3` maps to `gate1`, but Phase 4 Content Production tasks create approvals with a different `phaseNumber` that falls through to `null`, then defaults to `router.go('floor', ...)` — which itself then shows the brand review banner (gate1). More specifically, looking at the tower's `approvalItems` logic:

```js
const gateView = a.phaseNumber === 3 ? 'gate1'
  : a.phaseNumber === 6 ? 'gate2'
  : a.phaseNumber === 8 ? 'gate3'
  : null;
```

Phase 4 has no mapped gate view (returns `null`), so it falls back to `router.go('floor', ...)`. On the floor view, the `brandReady` banner then incorrectly shows `gate1` as the review destination because it reads from `pendingGate.phaseNumber` — but there is NO pending gate for phase 4 content production (it's a task-level approval, `a.type !== 'gate'`), so the non-gate branch routes to `router.go('floor', ...)` which loops. The real bug: the `viewFloorBuilding` bottom button and the gate-ready banner both compute `gateView` using only phaseNumbers 3/6/8 — any content production (phase 4/5) approval that uses `type !== 'gate'` is silently routed to `gate1` because `brandReady` is true (brand data is parsed) and the code defaults to `gate1`. The fix: add a `content-review` view and wire the Content Production task's Review button to it, plus guard the `brandReady` banner so it only fires for actual gate approvals.

After tracing more carefully: the `brandReady` flag is `!!pendingGate || brandDataParsed`. When `brandDataParsed` is true (which it will be after phase 3 completes), `brandReady` becomes `true` even in phase 4+. The gate banner then calls `router.go('gate1', ...)` because `pendingGate` is null (no gate-type approval exists for phase 4), so the ternary `pendingGate?.phaseNumber === 3 ? 'gate1' : ...` hits `null`, and the final fallback is `'floor'`... but the Review button at the bottom of `viewFloorBuilding` has an identical expression that resolves to `'floor'` when `pendingGate` is null, and the inline review button in the gate-ready banner uses `gateView` which also falls back to `'floor'`.

The **actual reported symptom** — Review goes to brand info screen — means the content production task approval IS appearing as a pending `gate`-type approval with `phaseNumber` that maps to `gate1` (phaseNumber === 3), OR the phase hasn't advanced past 3 locally. The most likely cause: after approving the foundation, `setLocalPhase(id, 4)` is called but the pending gate approval for phaseNumber 3 is **not removed from `state.approvals`** client-side, so the tower's `approvalItems` still routes to `gate1`. The banner in `viewFloorBuilding` also reads `pendingGate` from `state.approvals` and finds the stale phase 3 entry, sending the Review button to `gate1`.

The clean fix: in `viewFloorBuilding`, filter out gate approvals for phases that have already been passed (i.e., `phaseNumber <= phase - 1` should not be considered pending).

## Patch

### Change 1: Filter stale gate approvals in `viewFloorBuilding`

**What it changes:** When computing `pendingGate` in `viewFloorBuilding`, exclude any gate approval whose `phaseNumber` is for a phase the floor has already passed. This prevents a stale phase-3 gate approval from hijacking the Review button after the owner has already approved foundation and advanced to phase 4+.

**Why:** After `approveFoundation()` advances `currentPhase` to 4 locally, the backend approval record for phase 3 may still be `pending` in `state.approvals` until the next poll clears it. This stale entry causes every "Review" button on the floor to route to `gate1`.

File: `public/app.js`

```javascript

  // Exclude stale approvals for phases the floor has already passed (local phase override
  // may be ahead of the backend approval record between polls).

    a => a.floorId === floor.id

      && a.type === 'gate'
```
