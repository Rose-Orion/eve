## Diagnosis

The issue is in the backend, not the frontend routing. Looking at the `src/server/routes/floors.ts` file, the **Review button** on the content production page (Phase 5) routes to the content review flow. Tracing the full path: Phase 5 tasks complete → `checkPhaseCompletion` fires → `floor:phase-complete` event → `activateNextPhase` is called. However, `activateNextPhase` checks `current?.requiresGate` — and the Phase 5 phase definition's gate configuration determines whether the owner is routed to a gate approval page or auto-advances.

The deeper issue: the `/api/floors/:id/approve-gate/:phase` and `/api/floors/:id/approve-foundation` routes exist, but there is **no dedicated `/api/floors/:id/review-content` or `/api/floors/:id/approve-content` route** for Phase 5 content review. The "Review" button on the content production page likely hits the generic `approve-gate` endpoint, but the phase number being passed may be wrong (Phase 4 instead of Phase 5), or the gate approval response doesn't return enough state for the frontend to navigate to the correct next destination.

Let me trace more carefully: Phase 5 = Content Production. After Phase 5 completes, the gate leads to Phase 6 (Staging & QA). The Review button likely calls `POST /api/floors/:id/approve-gate/5`, which returns `{ success: true, nextPhase: 6 }` — but the frontend may be looking for a `floor` object with updated `status`/`currentPhase` to determine where to navigate next.

The `approvePhaseGate` method returns only a `boolean`, so the route returns `{ success: true, nextPhase: 6 }` with no floor state. The frontend can't navigate correctly because it doesn't know the new floor state.

## Patch

### Change 1: Return full floor state from approve-gate route so the frontend can navigate correctly

**What it changes:** The `approve-gate` route currently returns only `{ success: true, nextPhase }`. The frontend Review button needs the updated floor state (status, currentPhase, brandState) to navigate to the correct destination after approval. This adds the updated floor to the response.

**Why:** Without the floor object, the frontend has to guess where to navigate based on `nextPhase` alone — but if there's any mismatch between what the frontend expects and what the backend state is, navigation falls short or goes to the wrong page. Returning the full floor state makes the response self-describing.

### Change 3: Harden `approvePhaseGate` to return `false` only on actual failure, not on already-advanced gates

**What it changes:** Currently `approvePhaseGate` returns `true` if already past the gate — but it also returns `true` even if `phaseManager.approveGate` returns false (it logs a warning but proceeds). The route then calls `getFloor` after to retrieve state. This is fine. But the method needs to guarantee it returns `true` whenever the gate is effectively approved (including the already-past case), so the route never 404s on a valid idempotent re-click.

Tracing the "doesn't go all the way through" issue: `approvePhaseGate` calls `seedNextPhaseTasks` which has an async guard checking Supabase. If the DB check fails (throws), the task is never seeded and the method still returns `true` — so the route says success but the phase tasks weren't created. The navigation succeeds but the destination page shows no tasks.

File: `src/orchestrator/index.ts`

  /** Owner approves a phase gate (e.g. Foundation → Phase 4 Buildout). */
  async approvePhaseGate(floorId: string, phaseNumber: number): Promise<boolean> {
    const floor = this.floors.get(floorId);
    if (!floor) return false;

    // If already past this gate, return success without re-seeding
    if (floor.currentPhase > phaseNumber) {
      console.log(`[Floor] Gate ${phaseNumber} already approved for ${floor.name} (currentPhase=${floor.currentPhase})`);
      return true;
    }

    // Force-complete all prior phases so the gate can be approved
    this.phaseManager.forceCompleteUpTo(floorId, phaseNumber);
    const gateApproved = this.phaseManager.approveGate(floorId, phaseNumber);
    if (!gateApproved) {
      console.warn(`[Floor] approveGate returned false for phase ${phaseNumber} on ${floor.name} — phase may already be approved or in wrong state`);
      // Even if PhaseManager rejected, still advance if we haven't yet
    }

    // Phase 3 = Foundation Sprint gate: mark brand as approved so the UI clears the CTA
    if (phaseNumber === 3) {
      floor.brandState = 'foundation-approved';
      console.log(`[Floor] brandState → 'foundation-approved' for ${floor.name}`);
    }

    floor.currentPhase = phaseNumber + 1;

    // Persist floor state BEFORE seeding tasks so if seedNextPhaseTasks throws,
    // the floor's currentPhase is already correct and a retry will hit the
    // "already past this gate" guard above (idempotent). Without this ordering,
    // a seed failure leaves floor.currentPhase at the old value but the phase
    // manager state advanced — causing "doesn't go all the way through".
    const saved = await saveFloor(floor);
    if (!saved) {
      console.error(`[Floor] approvePhaseGate: saveFloor FAILED for "${floor.name}" (phase ${phaseNumber}) — reverting currentPhase`);
      floor.currentPhase = phaseNumber; // revert so retry is safe
      return false;
    }

    // Seed tasks for the next phase — errors are logged but don't fail the gate
    // approval (floor state is already persisted above). The owner can retry seeding
    // via the seed-foundation endpoint or by re-clicking Review if tasks are missing.
    try {
      await this.seedNextPhaseTasks(floorId, floor, phaseNumber);
    } catch (seedErr) {
      console.error(
        `[Floor] approvePhaseGate: seedNextPhaseTasks threw for phase ${phaseNumber + 1} ` +
        `on "${floor.name}": ${(seedErr as Error).message}. ` +
        `Floor phase advanced to ${floor.currentPhase} but tasks may not have been created. ` +
        `Owner should re-approve or use seed-foundation endpoint.`,
      );
      // Still broadcast so the frontend navigates — tasks page will show empty state
      // which is recoverable, vs. silently leaving the gate in a stuck state.
    }

    await broadcastFloorEvent(floorId, 'phase:advanced', {
      from: phaseNumber,
      to: phaseNumber + 1,
      brandState: floor.brandState,
    });

    // Broadcast brand state change so Tower re-renders without the Review CTA
    if (phaseNumber === 3) {
      await broadcastFloorEvent(floorId, 'brand:state-changed', {
        floorId,
        brandState: floor.brandState,
      });

      body: `${floor.name}: Phase ${phaseNumber + 1} tasks now queued.`,
      floorId,
      type: 'info',
    });
    console.log(`[Floor] Gate ${phaseNumber} approved — Phase ${phaseNumber + 1} starting for ${floor.name}`);
    return true;
  }
