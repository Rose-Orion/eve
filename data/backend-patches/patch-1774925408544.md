## Diagnosis

The Phase 6 (Staging & QA) dependency-trigger chain has three broken links: (1) `checkPhaseCompletion` is called with the completed task's phase number, but the `task:completed` event handler in `setupEventHandlers` does not re-check dependencies for tasks in *other* phases that may `dependsOn` Phase 6 tasks — because Phase 6 tasks have no `dependsOn` set, the `DependencyGraph.onTaskCompleted` cascade does nothing and the queue-to-next-phase path relies entirely on `checkPhaseCompletion` → `activateNextPhase` → `seedNextPhaseTasks`. (2) The `activateNextPhase` call from the `floor:phase-complete` event fires asynchronously but the event is emitted from `phaseManager.completePhase` which runs inside `checkPhaseCompletion` — if Phase 6 requires a gate (`requiresGate: true`), it goes to `gate-waiting` and the `floor:phase-complete` event is **never emitted**, so `activateNextPhase` is never called and Phase 7 is never seeded. This is working as designed for gate phases. (3) However the **real bug** is that `checkPhaseCompletion` is only triggered for the `task:completed` event and the `task:status-changed → escalated` path, but **not** when a task is restored to completed status during recovery (`restoreWithStatus`) — meaning floors recovered from the DB with all Phase 6 tasks already completed never re-trigger the gate check. Additionally, for the specific task types listed (`copy-review-redo`, `copy-review`, `staging-review`, etc.), there is **no `copy-review-redo` task type** defined anywhere in `seedNextPhaseTasks` case 6, meaning if a copy-review requires a redo pass, no downstream task is seeded to handle it.

The more critical runtime issue: in `setupEventHandlers`, the `task:completed` event listener calls `checkPhaseCompletion(data.floorId, completedTask.phaseNumber)` — but `completedTask` is fetched *after* `dependencyGraph.onTaskCompleted` runs. Since `taskManager.transition(task.id, 'completed')` fires the event *before* the orchestrator's listener runs (EventBus is synchronous), the task IS in the map when the listener runs. That part is fine.

The **actual broken link** is: `checkPhaseCompletion` checks `allDone` using `TERMINAL_STATUSES = ['completed', 'escalated']`, but there is no guarantee that all phase tasks are in-memory at check time for recovered floors. More critically, for Phase 6 specifically: the Phase 6 gate fires `approval:needed` (not `floor:phase-complete`), and the `activateNextPhase` handler only fires on `floor:phase-complete`. The `approveGate` method in PhaseManager emits `floor:phase-complete` after gate approval, which correctly seeds Phase 7. **The gate approval path is actually correct.**

The real bugs are: (a) No retry/re-check mechanism if `task:completed` events are missed (e.g., during server restart where tasks are restored as `queued` and re-dispatched), (b) the `copy-review-redo` task type referenced in the issue doesn't exist in Phase 6 seeding, and (c) there is no idempotency check ensuring that when Phase 6 tasks are all completed, `checkPhaseCompletion` is called even if the final `task:completed` event was missed.

The patch adds: a post-recovery phase completion sweep, a `copy-review-redo` task seeded after `copy-review` completes (dependency chain), and a periodic re-check in `processQueue` to catch stuck phases.
