## Diagnosis

The `task:completed` event handler in `setupEventHandlers()` triggers `checkPhaseCompletion()`, which in turn calls `dependencyGraph.onTaskCompleted()` to resolve newly-ready tasks. However, there is no structured logging in this path to confirm which tasks become ready after `budget-plan` completes, nor any logging to show why `brand-details` specifically is not being enqueued. Additionally, the `checkPhaseCompletion` call happens *after* dependency resolution, meaning if `brand-details` depends on `budget-plan`, the dependency cascade fires first — but without logging we cannot confirm the cascade is firing or that `brand-details` is registered in the dependency graph at all. The fix requires: (1) structured logging in the `task:completed` handler and `checkPhaseCompletion`, (2) logging in `seedNextPhaseTasks` to confirm `brand-details` is being created with correct `dependsOn` refs, and (3) a guard to ensure newly-created dependent tasks are registered in the dependency graph before the cascade fires.

        `${pending.length} task(s) still pending: ` +
        `[${pending.map(t => `${t.id.slice(0, 8)}(${t.taskType},${t.status})`).join(', ')}]`,
