## Diagnosis

The 'Review Content' task for floor `b090c7ea-ef9c-4bb3-abdf-62897e1224e8` persists on the dashboard after owner approval because the `handleApproval` flow in `TaskManager.setupListeners()` transitions the task to `'completed'` via the event bus, but there is no subsequent broadcast to Supabase realtime after that transition. The `task:completed` event fires internally, but the `dispatchVirtual` path that normally calls `broadcastFloorEvent(task.floorId, 'task:completed', ...)` is never reached — the approval path in `TaskManager` calls `transition()` directly, skipping the broadcast. Additionally, the dashboard reads `getPendingApprovals()` which checks `getFloorTasksByStatus(floor.id, 'review')` — if the task successfully transitions to `completed` in memory but Supabase doesn't get the realtime broadcast, the dashboard never re-queries and the stale review item persists.

There are two separate issues:
1. **Missing realtime broadcast** after task approval in the `TaskManager` `approval:received` handler — the transition to `completed` fires the internal `task:completed` event but not `broadcastFloorEvent`.
2. **No `review` → `completed` task save** — `ALWAYS_PERSIST_STATES` includes `completed`, so the DB write happens, but the realtime broadcast to the dashboard does not (unlike the `dispatchVirtual` path which explicitly calls `broadcastFloorEvent`).

The fix is in the Orchestrator's `setupEventHandlers` — add a `task:completed` handler that broadcasts to Supabase realtime and also handle the approval path to ensure the broadcast fires regardless of how completion is triggered.

## Patch

### Change 1: Broadcast `task:completed` to Supabase realtime on every task completion (not just via `dispatchVirtual`)

Currently `broadcastFloorEvent` for `task:completed` is only called inside `dispatchVirtual`. When a task is approved via `handleApproval`, the `TaskManager.setupListeners()` calls `transition(task.id, 'completed')` which emits the internal `task:completed` event — but the Orchestrator's `setupEventHandlers` handler for `task:completed` only handles dependency cascades and phase completion. It never broadcasts to Supabase realtime. This means the dashboard's realtime subscription never fires and the review item stays visible.

## Rollback

The changes are additive — they add broadcasts and reorder two lines. To revert:

1. **Change 1**: Remove the `broadcastFloorEvent` call added at the bottom of the `task:completed` handler.
2. **Change 2 & 3**: Revert the `approval:received` handler to start with `if (!String(data.taskId).startsWith('gate-')) return;` (removing the new non-gate block).
3. **Change 4**: Revert the `setupListeners` approval handler to its original order (move `task.reviewStatus = 'approved'` back to after `transition()`).

None of these changes modify security files, delete data, or alter the database schema. The worst-case scenario if something goes wrong is the broadcast fires an extra time (idempotent) or the task status save fires twice (also idempotent).
