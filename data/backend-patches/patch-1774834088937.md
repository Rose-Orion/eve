## Diagnosis

Two distinct persistence bugs identified:

1. **Brand name selection**: `saveFloor()` in `src/integrations/supabase.ts` does NOT include `selected_brand` or `theme_config` in the upsert payload. When `updateFloor()` calls `saveFloor(floor)` after setting `floor.selectedBrand`, the database write omits these fields entirely. Similarly, `loadFloors()` doesn't map `selected_brand` or `theme_config` back from the database rows — so even if somehow written, they'd be lost on reload.

2. **Chat history persistence**: The `loadChatHistory()` method has a guard `if (this.chatHistories.has(floorId)) return;` which is correct, but **the in-memory cache is never pre-populated on boot**. When the server restarts, `chatHistories` is empty — but messages ARE in Supabase. The first call to `getChatHistory()` or `sendChatMessage()` correctly loads from Supabase via `loadChatHistory()`. The real risk here is the **dual-write gap**: `persistChatMessage()` calls `saveChatMessage()` and catches errors silently, falling back to file — but if `saveChatMessage()` returns `null` (Supabase insert fails without throwing), the error isn't logged or handled. Also: `loadFloors()` returns floors without `selectedBrand` or `themeConfig`, so `sendChatMessage()` → `this.floors.get(floorId)` gets a floor with `selectedBrand: undefined`, which means `dispatchVirtual()` passes `null` for `selectedBrand` — silently breaking brand context in chat.

## Rollback

**Change 1 & 2 (supabase.ts — saveFloor/loadFloors):** The new fields use `?? null` so if the DB columns don't exist yet, Supabase will return a column-not-found error rather than silently failing — this is surfaced by the new `console.warn` in `saveFloor`. To rollback: revert the two functions to their original form (remove `selected_brand`, `theme_config` lines from the upsert, and remove the new fields from the loadFloors map). The schema columns can remain without harm.

**Change 3 (persistChatMessage):** Purely additive logging + null-check. To rollback: revert to the original `try/catch` that silently swallowed `null` returns. No data is at risk.

**Change 4 (loadChatHistory):** The key behavioral change is setting `this.chatHistories.set(floorId, [])` after a successful empty Supabase response (preventing unnecessary file reads). To rollback: remove the early `this.chatHistories.set(floorId, [])` call and the `if (this.chatHistories.has(floorId)) return;` guard that follows it, restoring the original flow.

**Change 5 (updateFloor brand selection):** The `saveFloor` call is now `await`ed with error logging instead of fire-and-forget. The `return floor` early exit after brand selection is new — if it causes issues (e.g., other updates in the same call being dropped), revert to the original `saveFloor(floor).catch(() => {})` at the end.

**Change 6 (checkConnection):** Purely additive — the new `colCheck` query is read-only and non-blocking. To rollback: remove the three lines after `if (error) return false;`.
