## Diagnosis

The bug is in the `task:completed` event handler in `setupEventHandlers()`. When `budget-plan` completes in Phase 1 (actually Phase 3 = Foundation Sprint), the `dependencyGraph.onTaskCompleted()` cascades dependents and `checkPhaseCompletion()` fires — but the **downstream tasks for the next phase are never seeded until the phase gate is approved** (`requiresGate: true` for Phase 3). This is correct behavior for the gate itself.

The real issue is: **after `budget-plan` completes in Phase 3, the other Phase 3 tasks (`brand-options`, `business-strategy`) must also complete before `checkPhaseCompletion` transitions to `gate-waiting`. But if only `budget-plan` completed and the others are missing or not queued, the phase never reaches `gate-waiting` — and the downstream Phase 4 tasks are never seeded.** More critically, if the Phase 3 tasks were seeded but the `dependsOn` relationships aren't wired correctly (they're all independent), there's no cascade trigger — each task's completion independently calls `checkPhaseCompletion`, which only advances the phase when **all** tasks are terminal.

The specific failure for Quest Kids Phase 1 (Phase 3): the investigation methods and budget-correction code handle `budget-plan` in isolation but **never check whether the other Phase 3 sibling tasks (`brand-options`, `business-strategy`) were seeded and queued after `budget-plan` completed**. If they were never created (or exist only as superseded/rejected), Phase 3 can never complete, so the gate never fires, so Phase 4 is never seeded.

      'and should not be used for downstream planning.';

    const viableTypes = new Set<string>();

          t.status === 'completed' &&

## Rollback

The changes are purely additive (new method + new event handler call + modified investigation branch). To roll back:

1. **Remove the `repairMissingPhase3Siblings` call** from the `task:completed` handler (Change 1) — revert to the original 4 lines.
2. **Delete the `repairMissingPhase3Siblings` method** (Change 2).
3. **Revert `investigatePhaseTaskQueue`** auto-remediation block to the original `seedFoundationTasks` call (Change 3).

No database schema changes, no data deletions, no security file modifications. All new tasks created by the repair are standard `TaskManager.create()` calls — they can be manually deleted from Supabase if needed (`DELETE FROM tasks WHERE floor_id = '...' AND phase_number = 3 AND task_type IN ('brand-options', 'business-strategy') AND status = 'queued'`).
