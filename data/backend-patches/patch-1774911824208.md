## Diagnosis

The stuck copy-review task needs retries paused immediately. The core issue is that `retryTask()` clears `task.attempts = 0` and `task.result = null` but never persists rejection notes from prior slop-violation failures — `reviewFeedback` is nulled implicitly via `task.result = null` (which doesn't touch `reviewFeedback`), but the rebuilt prompt in `rebuildTaskPrompt()` and the sibling-context injection in `dispatchVirtual()` contain **no mechanism to forward prior rejection notes into the next retry's prompt**. Each retry starts fresh without seeing what failed before.

   * Pause all further retries on a task immediately.
   * Sets a sentinel reviewFeedback flag that recordFailure() checks before scheduling
   * the next retry. Does NOT change task status — the task stays in its current state
   * so the owner can inspect it. Returns false if task not found.

## Confirmation Report

**What was found:**

1. **Retry pausing:** No mechanism existed to immediately halt retries on a stuck task. The owner had no way to stop the cycle without manually escalating or deleting the task. ✅ Fixed with `pauseRetries()`.

2. **Full task context retrieval:** No single endpoint exposed both the raw task object AND the constructed agent instruction payload together. ✅ Fixed with `GET /api/tasks/:taskId/context`.

3. **Feedback passthrough inspection:** The `reviewFeedback` field on the Task is the correct persistence location for rejection notes. However, the slop-violation failure path was **overwriting** `reviewFeedback` on each retry call to `recordFailure()` (via indirect clearing — actually `reviewFeedback` wasn't being set at all in the slop path, it remained `null` from task creation). The `dispatchVirtual` code called `recordFailure(task.id, 'Anti-slop: rewrite without...')` but the failure reason went only into the `recordFailure` error string, never into `task.reviewFeedback`. **Result: rejection notes were being silently dropped.** ✅ Fixed by accumulating notes into `task.reviewFeedback` in the slop check block.

4. **Feedback injection on retry:** Even if notes were persisted into `reviewFeedback`, no code path read them back and injected them into the next retry's prompt. The `dispatchVirtual` rebuilt the prompt using sibling context and `task.prompt` but ignored `task.reviewFeedback` entirely. **Result: copy-agent received zero feedback about what it did wrong on prior attempts.** ✅ Fixed by prepending the accumulated rejection notes to `description` before dispatch.
