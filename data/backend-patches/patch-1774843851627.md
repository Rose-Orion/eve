## Diagnosis

There are three compounding bugs causing the Tower to keep showing "Review →" after foundation approval:

1. **`approvePhaseGate()` saves `floor.brandState` as `'pre-foundation'`** — it only updates `currentPhase` and calls `saveFloor()`, but never sets `floor.brandState = 'foundation-approved'`. The Tower CTA almost certainly checks `brandState === 'foundation-review'` (or similar), so approval never clears that flag in the DB.

2. **`approveGate()` in `PhaseManager` requires `phase.status === 'gate-waiting'`**, but if the phase was previously force-completed or already transitioned, it silently returns `false` — meaning the phase may not actually advance even when `approvePhaseGate()` thinks it succeeded (it doesn't check the return value of `approveGate()`).

3. **The `approval:received` event handler in `setupEventHandlers()` duplicates the gate-approval logic** without updating `brandState` either, and both paths can run concurrently when `handleApproval()` is called, causing the "approved twice" symptom: first call goes through `approvePhaseGate()` directly; second call fires the event handler. Neither sets `brandState`.

The fix: (a) update `brandState → 'foundation-approved'` in `approvePhaseGate()` whenever phase 3 is approved, (b) broadcast a `brand:state-changed` event so the UI can react, and (c) add a guard in the `approval:received` event handler to prevent double-execution when `approvePhaseGate()` was called directly.

  approveGate(floorId: string, phaseNumber: number): boolean {
    const phases = this.floors.get(floorId);
    if (!phases) return false;

    const phase = phases[phaseNumber - 1];
    if (!phase) return false;
