## Diagnosis

**Part 1:** The `brand-details` task for Quest Kids (Phase 1) needs to be manually enqueued immediately. Based on the codebase, Phase 1 of Quest Kids appears to be Phase 3 (Foundation Sprint) — the `budget-plan` task completed but `brand-details` is not a standard task in the current pipeline. **Part 2:** The root cause of the inconsistent dependency-trigger-to-queue pathway is a **missing `brand-details` entry in the `seedNextPhaseTasks` switch statement** combined with a **timing/scope issue in `setupEventHandlers`** where the `task:completed` handler evaluates `checkPhaseCompletion` but the phase transition only seeds tasks defined in the switch — any unlisted task type simply never gets created.

Additionally, there's a secondary issue: `setupEventHandlers()` is called in the constructor, which is correct (global, once at boot), but the `task:completed` handler reads `data.floorId` which only exists if the event was emitted with that field — checking the `eventBus.emit('task:completed', ...)` in `transition()`, it passes `floorId` correctly. The real problem for the inconsistency is that **`brand-details` is simply absent from the dependency map / task-type routing table in `seedNextPhaseTasks`**.

    floorIdOrName: string,
    taskType: string,

      phaseNumber?: number;
      assignedAgent?: AgentId;

      return { success: false, message: `Floor not found: ${floorIdOrName}` };
    }

      'brand-details':        'brand-agent',
      'brand-options':        'brand-agent',
      'brand-visual-system':  'design-agent',

    const priority = options.priority ?? 'high';

    // Build a default prompt from completed deliverables if none provided
    let prompt = options.prompt ?? '';

      const allTasks = this.taskManager.getFloorTasks(floor.id);
      const completedResults = allTasks

        .join('\n\n');

        ? `\nOWNER-SELECTED BRAND: ${sb.name} — "${sb.tagline}" (${sb.personality})`

        `${brandCtx}\n\n` +

    const task = this.taskManager.create({
      floorId: floor.id,
      phaseNumber,
      assignedAgent,
      modelTier: this.modelRouter.getModelTier(assignedAgent, 'foundation'),
      taskType,
      description,
      prompt,
      priority,
    });

    this.dependencyGraph.addTask(task.id, task.dependsOn);

    }).catch(() => {});

      floorName: floor.name,
      taskId: task.id,
      taskType,

      assignedAgent,
      priority,

      `for "${floor.name}" (phase ${phaseNumber}, agent: ${assignedAgent}, priority: ${priority})`,

      body: `"${taskType}" task enqueued for "${floor.name}" (phase ${phaseNumber}). Will dispatch in next cycle.`,
      floorId: floor.id,

      success: true,
      taskId: task.id,
      message: `Task "${taskType}" enqueued as ${task.id.slice(0, 8)} for "${floor.name}" (phase ${phaseNumber}, agent: ${assignedAgent}, priority: ${priority}). Will dispatch in next processing cycle.`,

  /**
   * POST /api/floors/:floorId/enqueue-task
   * Body: { taskType: string, phaseNumber?: number, assignedAgent?: string,
   *         priority?: string, description?: string, prompt?: string }
   *
   * Manually enqueues a task for a floor to unblock a stalled pipeline.
   * Idempotent: creates a new task even if one of the same type exists
   * (the orchestrator will dispatch the highest-priority queued task first).
   */
  app.post<{
    Params: { floorId: string };
    Body: {
      taskType: string;
      phaseNumber?: number;
      assignedAgent?: string;
      priority?: string;
      description?: string;
      prompt?: string;
    };
  }>('/api/floors/:floorId/enqueue-task', async (request, reply) => {
    const { floorId } = request.params;
    const { taskType, phaseNumber, assignedAgent, priority, description, prompt } = request.body ?? {};

    if (!taskType || typeof taskType !== 'string') {
      return reply.code(400).send({ error: 'taskType is required' });
    }

    const result = await orchestrator.enqueueTask(floorId, taskType, {
      phaseNumber,
      assignedAgent: assignedAgent as any,
      priority: priority as any,
      description,
      prompt,
    });

    return reply.code(result.success ? 200 : 404).send(result);
  });

## Root Cause Report

### Why the trigger fires inconsistently

**Primary cause: Missing task type in `seedNextPhaseTasks`**

The `task:completed` → `checkPhaseCompletion` → `activateNextPhase` → `seedNextPhaseTasks` chain works correctly. The handler is registered globally at boot (constructor → `setupEventHandlers()`), once, before any floor is initialized — so there is no registration timing issue.

The real failure is that `brand-details` was never in the `case 4` block of `seedNextPhaseTasks`. When Phase 3 (`budget-plan`) completed and Phase 4 was seeded, `brand-details` was simply not created. You cannot auto-enqueue a task that was never seeded.

**Secondary cause: No fallback for unlisted task types**

The switch has a `default: console.log('No task template')` branch that silently does nothing. Any task type not listed in the switch is permanently absent. There is no error, no audit log entry, no notification — it's a silent skip.

**Why it's inconsistent across floors**

- Floors where Phase 3 → Phase 4 completed normally got `brand-visual-system` and `brand-voice-guide` but never `brand-details` (it was never in the pipeline for any floor).
- Quest Kids hit this gap more visibly because the `budget-plan` correction work surfaced it — the foundation sprint completed but the expected next task in the owner's mental model (`brand-details`) was absent from the system.

**Answers to the specific investigation questions:**

1. **Event listener timing**: No issue. `setupEventHandlers()` runs in the constructor before `loadPersistedState()` or `start()`. The handler is global, registered once. There is no per-floor listener.

2. **Handler scope**: No issue. The handler reads `data.floorId` from the event payload (correctly passed in `transition()` → `eventBus.emit('task:completed', {taskId, floorId, ...})`), then calls `checkPhaseCompletion(data.floorId, ...)` which is correctly scoped.

3. **Race condition / partial patch**: No race condition in the handler itself. The prior budget-plan work added a supersession guard in `checkPhaseCompletion` (phase 3 blocked until valid `$500` task completes) — this is correct and does not affect phase 4 seeding. The gap is purely the missing `brand-details` entry.

**The fix is durable**: Change 2 adds `brand-details` to `case 4` of `seedNextPhaseTasks` permanently, so all future floors of any type will have it seeded. Change 1 adds `enqueueTask()` for any future gaps. The fix covers all floor types (`isEcommerce` check is not needed for `brand-details` — it applies to all business types).

**Double-enqueue protection**: The existing guard at the top of `seedNextPhaseTasks` prevents re-seeding:
```typescript
const existingPhaseTasks = this.taskManager.getFloorTasks(floorId).filter(
  t => t.phaseNumber === nextPhase && t.status !== 'completed' && t.status !== 'escalated',
);
if (existingPhaseTasks.length > 0) { ... return; }
```
This means floors where Phase 4 was already seeded (with `brand-visual-system` and `brand-voice-guide`) will not be re-seeded or double-enqueued on server restart.
