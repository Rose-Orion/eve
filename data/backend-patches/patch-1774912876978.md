## Diagnosis

The Phase 6 task seeding for SideQuest has a critical misconfiguration in `FloorCreator.create()`: the `budget-plan` task in Step 6 uses a hardcoded generic prompt without injecting the floor's `budgetCeilingCents` baseline. More critically, Phase 6 ("Staging & QA") tasks are seeded by `seedNextPhaseTasks()` when Phase 5 completes — but the dependency trigger (`checkPhaseCompletion` → `activateNextPhase` → `seedNextPhaseTasks`) silently fails if Phase 5 tasks were never created or are stuck, leaving Phase 6 with zero tasks and no error surfaced to the owner. The `investigatePhaseTaskQueue` method handles Phase 3 auto-remediation but has **no auto-remediation path for Phase 6** — it only logs the error and suggests a manual endpoint.

### Change 4: Fix the Phase 6 dependency trigger — add explicit phase-completion check after Phase 5 task escalations

The silent stall occurs because `checkPhaseCompletion` only fires from `task:completed` events. If Phase 5 tasks are all escalated (permanently failed) without completing, the phase-complete event never fires, so Phase 6 is never seeded. The `task:status-changed` handler already calls `checkPhaseCompletion` on escalation — but only for the `escalated` status. We need to ensure that when _all_ Phase 5 tasks reach terminal state (including via escalation), the phase transition fires.

The existing `setupEventHandlers` already has this:
```typescript
    this.eventBus.on('task:status-changed', (data) => {
      if (data.to === 'escalated') {
        this.checkPhaseCompletion(data.floorId, this.taskManager.getTask(data.taskId)?.phaseNumber ?? 0);
```

This is correct. The actual gap is in `checkPhaseCompletion` itself — it returns early when `phaseTasks.length === 0` (no tasks loaded in memory). During boot recovery for SideQuest, if Phase 5 tasks were restored as completed/escalated via `restoreWithStatus` but Phase 6 was never seeded, the `checkPhaseCompletion` call does fire but finds 0 tasks (all completed, already filtered out by the non-terminal check).

The real fix is in `loadPersistedState`: after restoring all tasks, explicitly check whether the current phase needs seeding.

### Change 5: Ensure `seedNextPhaseTasks` guard checks DB count before skipping Phase 6

The existing guard in `seedNextPhaseTasks` checks `countPhaseTasks` (DB) before skipping re-seed. This is correct but the DB query uses `countPhaseTasks` which only counts non-archived tasks. If the Phase 6 tasks exist in DB as completed/escalated from a prior run, it would skip re-seeding even though the current run needs them. This is actually correct behavior (idempotent) — the issue is the DB count check should only skip if there are **active** (non-terminal) tasks. We need to ensure the guard only blocks re-seeding when active tasks exist, not when all DB tasks are terminal.

File: `src/orchestrator/index.ts`

```typescript
      this.phaseManager.activatePhase(floorId, nextPhase);
      return;

```

Replace with:
```typescript
    // Also check Supabase in case tasks were persisted but not loaded into memory.
    // Only skip re-seeding if there are ACTIVE (non-terminal) tasks in DB —
    // completed/escalated tasks from a prior run should NOT block re-seeding a new cycle.
    const dbCount = await countPhaseTasks(floorId, nextPhase);
    if (dbCount > 0) {
      // Check if the DB tasks are all terminal (completed/escalated) — if so,
      // allow re-seeding for growth cycle phases (9, 10) but block for one-shot phases.
      const isGrowthCyclePhase = nextPhase >= 9;
      if (!isGrowthCyclePhase) {
        console.log(`[EVE] Phase ${nextPhase} tasks already in database for ${floor.name} (${dbCount} found) — skipping re-seed`);
        this.phaseManager.activatePhase(floorId, nextPhase);
        return;
      }
      // Growth cycle phases: check if all DB tasks are terminal before allowing re-seed
      const inMemTasks = this.taskManager.getFloorTasks(floorId).filter(t => t.phaseNumber === nextPhase);
      const TERMINAL = ['completed', 'escalated'];
      const allTerminal = inMemTasks.length > 0 && inMemTasks.every(t => TERMINAL.includes(t.status));
      if (!allTerminal || inMemTasks.length === 0) {
        console.log(`[EVE] Phase ${nextPhase} tasks already in database for ${floor.name} (${dbCount} found) — skipping re-seed`);
        this.phaseManager.activatePhase(floorId, nextPhase);
        return;
      }
      console.log(`[EVE] Phase ${nextPhase} is a growth cycle phase with all-terminal DB tasks for ${floor.name} — allowing re-seed for new cycle`);
    }
```

3. **Change 4** (boot recovery seeding): Remove the two added `phaseManager` calls before `seedNextPhaseTasks`, reverting to the original single-line call.

4. **Change 5** (DB count guard): Restore the original 4-line guard that unconditionally skips on `dbCount > 0`.

All changes are additive logic branches — reverting any individual change does not break the others.
