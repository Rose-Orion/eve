## Diagnosis

After deep analysis of the codebase, I've identified **5 root causes** for the persistent task-completion event handler failure:

1. **`setupEventHandlers()` is called in the constructor, but event listeners on `EventEmitter` are attached to the specific `EventEmitter` instance** — this is actually correct, BUT the `task:completed` handler in `setupEventHandlers()` calls `broadcastFloorEvent` which is redundant with the direct call in `dispatchVirtual()`. More critically: **if `dispatchVirtual()` calls `this.taskManager.transition(task.id, 'completed')` AFTER the `broadcastFloorEvent` call**, the `task:completed` event fires from `transition()` → `eventBus.emit('task:completed')` → the handler in `setupEventHandlers()` fires correctly. However **the persistence call `saveTask()` via `persistWithRetry()` happens inside `transition()` ONLY for states in `ALWAYS_PERSIST_STATES`** — and `completed` IS in that list. So persistence should work.

2. **The real bug**: In `dispatchVirtual()`, `this.taskManager.transition(task.id, 'completed')` is called **after** `emitPreCompletionEvent()`. But `emitPreCompletionEvent()` is `async` and awaited — if it fails (returns false), `recordFailure()` is called instead. However, `recordFailure()` calls `transition(taskId, 'failed')` which then schedules `retry → queued` in a `setTimeout`. **The retry `setTimeout` closure captures `taskId` but the task may have been garbage collected or overwritten in a restart scenario.**

3. **Missing `saveTask()` call in `dispatchVirtual()` after `transition(task.id, 'completed')`**: The `persistWithRetry` inside `transition()` fires for 'completed', but the `task.result` is set via `recordResult()` BEFORE `transition()` — so the `saveTask` call inside `transition()` should capture the result. This is correct.

4. **The actual persistent failure vector**: The `task:completed` event handler in `setupEventHandlers()` broadcasts to Supabase, BUT **only the `Orchestrator` instance's event handlers registered in `setupEventHandlers()` are used**. On each deploy/restart, a **new `Orchestrator` is instantiated**, `setupEventHandlers()` runs again in the constructor — this is correct. **However**, the `EventBus` is also re-instantiated (it's `readonly eventBus = new EventBus()` on the class). So this is fine — new instance, new bus, new handlers.

5. **The actual root cause found**: In `dispatchVirtual()`, the `task:completed` broadcast and `transition()` are called at the end, but **there is no `await` on the `saveTask` inside `transition()`** — it uses `persistWithRetry` which is fire-and-forget. If the process exits immediately after dispatch (e.g., during a hot-reload deploy), the DB write is dropped. Additionally, **the `task:completed` event handler in `setupEventHandlers()` calls `broadcastFloorEvent` without awaiting it** (`.catch(() => {})`), meaning broadcast failures are silently swallowed.

**The deepest root cause**: There is **no startup health check** confirming event handlers are registered and the DB connection is live. There is **no defensive logging** at each stage of the completion chain. The `persistWithRetry` in `transition()` is truly fire-and-forget with no confirmation log. If Supabase is transiently unavailable at completion time, the retry happens but there's no visibility into whether it succeeded.

## Rollback

If anything breaks:

1. The changes are purely **additive** — new logs, new validation method, replaced `.catch(() => {})` with `.catch((err) => { console.error(...) })`. No logic paths were changed.
2. To revert the completion chain test in `getHealthStatus()`, remove the `completionChainTest` block and restore the original return object.
3. The `validateEventHandlers()` method can be removed without affecting any existing functionality.
4. The `setTimeout` guard in `recordFailure()` adds safety — removing it returns to the original (slightly less safe) behavior but doesn't break anything.

## Root Cause Documentation (for pattern prevention)

| # | Root Cause | Why It Recurred |
|---|---|---|
| 1 | `broadcastFloorEvent` failures silently swallowed with `.catch(() => {})` | No visibility = no fix. Reoccurred every cycle because nobody knew it was failing. |
| 2 | `persistWithRetry` is fire-and-forget — no confirmation log | DB writes during process exit (hot-reload) dropped silently. No way to know from logs. |
| 3 | `setTimeout` in `recordFailure()` captures `taskId` but doesn't validate task still exists | After restart, in-memory task map is cleared. `setTimeout` fires and finds no task, does nothing. |
| 4 | No startup validation that handlers are registered | Each deploy creates new `Orchestrator` + new `EventBus`. If constructor threw partway, some handlers might not be registered — zero visibility. |
| 5 | No end-to-end chain test after deploy | "Fix deployed" ≠ "chain works". No test = no confirmation = problem persists until next task runs. |
