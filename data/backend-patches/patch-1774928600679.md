## Diagnosis

The issue has three components: (1) the `task:completed` event handler in `setupEventHandlers` does **not** check whether the just-completed task's phase completion triggers enqueuing of dependent tasks for Phase 1's `brand-details` task — but more critically, (2) there is a race condition in `checkPhaseCompletion` for Phase 3 where the supersession guard can block phase completion even when the `budget-plan` task has completed correctly, and (3) the `brand-details` task type is not seeded anywhere in `seedNextPhaseTasks` (Phase 4 seeds `brand-visual-system` and `brand-voice-guide` but not `brand-details`), meaning if the workflow expects a standalone `brand-details` task it would never be enqueued.

Given the issue is specifically about **Quest Kids Phase 1** (which maps to Phase 3 Foundation Sprint in the codebase), the primary fix needed is: ensure that when `budget-plan` completes, `checkPhaseCompletion` correctly evaluates all Phase 3 tasks as terminal and triggers the `floor:phase-complete` event, which then calls `activateNextPhase` → `seedNextPhaseTasks(phase 4)`. The race condition is that `checkPhaseCompletion` is called from the `task:completed` event handler, but the `task.status` may not yet be `'completed'` at the point the handler fires because `transition('completed')` emits `task:completed` **and then** `checkPhaseCompletion` reads back the task status — this is actually fine. However, there's a subtler issue: the `task:completed` event handler in `setupEventHandlers` receives `data.taskId` and calls `this.taskManager.getTask(data.taskId)` to get the `phaseNumber`, but if `brand-details` is expected as a Phase 3 task it simply doesn't exist in the task definitions.

After careful reading: **`brand-details` is not a task type defined anywhere in Phase 3 or Phase 4 seeding**. The Phase 3 Foundation Sprint seeds `brand-options`, `business-strategy`, and `budget-plan`. Phase 4 seeds `brand-visual-system`, `brand-voice-guide`, and optionally `product-catalog`. There is no `brand-details` task. The issue says "confirm whether the brand-details task exists — if missing, re-enqueue it with correct phase/floor context." Given the task name, it most likely belongs in **Phase 4** alongside `brand-visual-system` and `brand-voice-guide`, assigned to `brand-agent`.

The fix: (1) add `brand-details` task seeding to Phase 4 in `seedNextPhaseTasks`, (2) add a repair method to enqueue a missing `brand-details` task for Quest Kids, and (3) ensure the `task:completed` handler correctly triggers dependency cascade so `brand-details` dispatches after Phase 3 completes.
