## Diagnosis

The `queue-status` event is being emitted correctly on the internal `EventBus`, but the `task:queue-status` handler in `setupEventHandlers()` calls `broadcastFloorEvent` to push it to Supabase realtime — however, the feedback chain subscription that floor-manager verification loops depend on is receiving the event under `'task:queue-status'` channel name, while the `floor-manager-feedback` broadcast (which carries the three required fields: `task_id`, `current_queue_state`, `execution_outcome`) is only emitted for `budget-plan` tasks assigned to `finance-agent`. The core issue is that the `task:queue-status` EventBus event handler does **not await** `broadcastFloorEvent`, so if the Supabase realtime broadcast fails silently, the feedback chain never receives the event and the verification loop exhausts its cycles. Additionally, the `floor-manager-feedback` channel broadcast is inside the handler but missing error handling, meaning a single failure breaks the whole chain without any retry or fallback logging.
