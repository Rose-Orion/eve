## Diagnosis

The issue requires a structured audit of: (1) whether a `brand-details` task was enqueued after the prior dependency-trigger fix, (2) why it's not visible in the task queue (silent failures, filtering bugs, orphaned records), (3) whether the event-handler fix persisted, (4) whether the dependency-trigger logic is wired correctly for Quest Kids, and (5) manual re-trigger capability.

Reading the codebase, I can identify the root cause: **there is no `brand-details` task type defined anywhere in the orchestrator**. The phase seeding logic in `seedNextPhaseTasks()` only creates `brand-options`, `brand-visual-system`, and `brand-voice-guide` tasks. The `investigatePhaseTaskQueue()` method searches for brand tasks using a filter that includes `brand-details` in `BRAND_TASK_TYPES`, but no task of this type is ever created. Additionally, there is no dedicated audit endpoint to inspect whether the dependency-trigger fix persisted for a specific floor, and no way to manually re-trigger a `brand-details` task enqueue through the existing API surface.

The fix requires: (1) a new `auditDependencyTriggerFix` method that checks event handler wiring, (2) a `enqueueBrandDetailsTask` method to manually create and register the task, and (3) hooking these into the public API surface of the Orchestrator.
