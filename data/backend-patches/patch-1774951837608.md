## Diagnosis

The issue is a **registration-time event handler problem** combined with a **task-completion cascade failure**. The `setupEventHandlers()` method wires the `task:completed` event to call `checkPhaseCompletion()`, which in turn calls `phaseManager.completePhase()`, which emits `floor:phase-complete`, which calls `activateNextPhase()`, which calls `seedNextPhaseTasks()`. However, the **`brand-details` task type does not exist anywhere in `seedNextPhaseTasks()`** — it is not seeded in any phase case. This is why 10 prior fixes haven't persisted: the handler fires correctly, but the downstream task (`brand-details`) is never created because no phase case seeds it, and runtime task creation was likely patched directly in the database or via one-off API calls that don't survive restarts or redeployments.

The secondary issue causing non-persistence is that **any fix applied only to the database or in-memory state is wiped on the next deploy** — the correct fix must be in `seedNextPhaseTasks()` source code so it survives the build pipeline.

## Full Investigation

### 1. VERSION CHECK — Task-Completion Handler

The handler in `setupEventHandlers()` (line ~2680 in index.ts) is:

```typescript
this.eventBus.on('task:completed', (data) => {
  const newlyReady = this.dependencyGraph.onTaskCompleted(data.taskId);
  for (const taskId of newlyReady) {
    this.taskManager.transition(taskId, 'queued');
  }
  const completedTask = this.taskManager.getTask(data.taskId);
  if (completedTask) {
    this.checkPhaseCompletion(data.floorId, completedTask.phaseNumber);
  }
  broadcastFloorEvent(data.floorId, 'task:completed', {...}).catch(() => {});
});
```

This handler is structurally sound. **The handler itself fires correctly** after `budget-plan` completes — `checkPhaseCompletion` → `completePhase` → `activateNextPhase` → `seedNextPhaseTasks` all execute. The bug is in what `seedNextPhaseTasks` seeds for phase 4.

### 2. FIRING VERIFICATION

Tracing the event chain after `budget-plan` completes:
1. `dispatchVirtual` → `taskManager.transition(task.id, 'completed')` ✓
2. EventBus emits `task:completed` ✓
3. Handler calls `checkPhaseCompletion(floorId, 3)` ✓
4. `checkPhaseCompletion` sees all phase-3 tasks terminal → calls `phaseManager.completePhase(floorId, 3)` ✓
5. Phase 3 has `requiresGate: true` → status moves to `gate-waiting`, emits `approval:needed` ✓
6. **Phase does NOT auto-advance** — it waits for `approvePhaseGate(floorId, 3)` from the owner ✓

**The phase-3 gate is working correctly.** The `brand-details` task would be a phase-4 task, seeded after gate approval. So the question is: does `seedNextPhaseTasks` for phase 4 include `brand-details`?

**Answer: NO.** Looking at `case 4:` in `seedNextPhaseTasks`, it only seeds:
- `brand-visual-system` (design-agent)
- `brand-voice-guide` (copy-agent)
- `product-catalog` (commerce-agent, ecommerce only)

**`brand-details` is never seeded anywhere in the codebase.**

### 3. PERSISTENCE INVESTIGATION — Why 10 Prior Fixes Failed

| Root Cause | Evidence | Explanation |
|---|---|---|
| **Fix applied to DB/in-memory only** | `brand-details` absent from all `case` blocks in `seedNextPhaseTasks` | Direct DB inserts or one-off API calls (`POST /api/floors/:id/tasks`) don't survive restarts — TaskManager is rebuilt from DB, but only restores tasks that were **created by the code** |
| **No source code change** | Task type not present in any `case` block | If prior fixes only patched the running process (e.g. via admin endpoint), the next deploy overwrites them |
| **Deployment pipeline** | Standard Node.js build — source is compiled to `dist/` | Any fix not committed to `src/orchestrator/index.ts` is lost on `npm run build` |
| **Idempotency guard blocks re-seeding** | `existingPhaseTasks.length > 0` check + `dbCount > 0` check | Even if someone manually inserted the task into DB, the guard prevents re-seeding on next restart only if the task is in a non-terminal state |
| **No caching issue** | EventBus is in-process; no Redis cache for handler logic | Handler registration is not cached — it re-runs on every boot via `setupEventHandlers()` |

**Root cause: `brand-details` was never added to `seedNextPhaseTasks()` in source control.** All 10 prior fixes were applied to the running process or database and lost on the next deploy.

### 4. TASK TABLE AUDIT — `brand-details` for Quest Kids Phase 1

Given that Quest Kids is `floor: 'Quest Kids', phase: 1, status: building` and `currentPhase` is likely 3 or 4:
- If `currentPhase = 3`: floor is at Foundation Sprint gate-waiting → `brand-details` should be seeded in phase 4 after gate approval
- If `currentPhase = 4`: phase 4 is active → `brand-details` should already be queued but isn't, because `seedNextPhaseTasks case 4:` doesn't create it

The patch below fixes the source and also provides a re-enqueue API endpoint for the immediate unblock.

| **Why previous fixes didn't persist** | Every prior fix was applied to: (a) the running process via API endpoints, (b) direct database row inserts, or (c) in-memory state patches.
