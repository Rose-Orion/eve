## Diagnosis

The finance-agent's `budget-plan` task completion event is not triggering the downstream `brand-details` progression because: (1) the `task:completed` event handler in `setupEventHandlers()` calls `checkPhaseCompletion()` which checks if ALL phase 3 tasks are terminal — but the gate check at line ~2070 blocks phase 3 completion if *all* budget-plan tasks are marked superseded (`reviewStatus='rejected'`), even when a newly completed $500 task exists alongside them; (2) the `checkPhaseCompletion` method's supersession gate only checks for budget-plan tasks with `reviewStatus !== 'rejected'`, but after `invalidateAndRequeueBudgetPlan()` runs it marks ALL completed budget-plan tasks as superseded — if the new $500 task then completes and `recordResult` + `transition('completed')` fires, it correctly has `reviewStatus='pending'` (not rejected), so that gate *should* pass. The actual failure is more subtle: in `dispatchVirtual`, the `taskManager.transition(task.id, 'completed')` call at the very end fires the `task:completed` EventBus event, but the `task:completed` event registered in `setupEventHandlers` calls `this.dependencyGraph.onTaskCompleted(data.taskId)` — however `brand-details` is a **phase 4** task type (`brand-visual-system`, `brand-voice-guide`) that only gets seeded *after* `checkPhaseCompletion` fires `floor:phase-complete` → `activateNextPhase` → `seedNextPhaseTasks`. The real bug is that **`checkPhaseCompletion` is never being called with the correct phase number** when the budget-plan task completes because `data.taskId` resolves to the task but the `task:completed` event handler reads `completedTask.phaseNumber` — if the `task` was transitioned to `completed` but `getTask(data.taskId)` returns `undefined` (race condition: `dispatchVirtual` calls `transition` which emits `task:completed`, but the handler runs synchronously and `getTask` should find it), OR the supersession gate incorrectly treats the newly-completed $500 task as superseded because `recordResult` sets `task.result` but does NOT clear `reviewFeedback` which was set during the `emitPreCompletionEvent` retry hold logic.

After tracing the full code path: `emitPreCompletionEvent` sets `task.reviewFeedback = "Held in pending_verification..."` on retry attempts. If the emission succeeds on attempt 2 or 3, `reviewFeedback` is LEFT with that stale content. Then `checkPhaseCompletion`'s supersession gate checks `t.reviewFeedback === SUPERSESSION_NOTE` — so it won't falsely block. But there's another issue: `dispatchVirtual` calls `this.taskManager.transition(task.id, 'completed')` at the very bottom — but `transition('working' → 'completed')` is NOT in the valid transitions map! Valid from `working` is `['review', 'completed', 'failed']` — `completed` IS valid. So that's fine.

The **actual root cause**: In `dispatchVirtual`, after `emitPreCompletionEvent` returns `true`, the code calls `this.taskManager.transition(task.id, 'completed')` — but `transition` emits `task:completed` synchronously, which calls `checkPhaseCompletion`. Inside `checkPhaseCompletion`, `phaseTasks` is fetched via `getFloorTasks(floorId)` which returns all tasks including the budget-plan task. The supersession check at phase 3 looks for `validCompletedBudgetPlan` — a task that is `status === 'completed'` AND NOT (`reviewStatus === 'rejected'` AND `reviewFeedback === SUPERSESSION_NOTE`). The new $500 task should pass this. **BUT** — `transition` is called on `task.id` (the local variable), and `task` was passed by reference into `dispatchVirtual`. The task's status was already set to `'working'` before dispatch. After `recordResult`, `task.result` is set. After `transition('completed')`, `task.status` = `'completed'` and `task.completedAt` is set. This should work.

Re-reading more carefully: the `task:completed` event handler in `setupEventHandlers` does:
```js
const completedTask = this.taskManager.getTask(data.taskId);
if (completedTask) {
  this.checkPhaseCompletion(data.floorId, completedTask.phaseNumber);
}
```

`getTask` looks up `this.tasks.get(taskId)` — the task IS in the map (it was put there at creation). After `transition('completed')`, `task.status = 'completed'`. So `checkPhaseCompletion` is called with the correct phase number.

The real issue is **silent error swallowing in `activateNextPhase`**. When `floor:phase-complete` fires → `activateNextPhase` is called → `seedNextPhaseTasks(floorId, floor, completedPhase)` is called. If ANY error occurs in `seedNextPhaseTasks` (e.g. `countPhaseTasks` throws, or Supabase is unavailable), the `.catch(() => {})` on `activateNextPhase` silently swallows it, and the phase 4 tasks are NEVER seeded. Additionally, `activateNextPhase` checks `if (current?.requiresGate) return;` — Phase 3 IS a gate phase (`requiresGate: true`), so `activateNextPhase` **returns early without seeding phase 4 tasks**! This is the root cause: the `floor:phase-complete` handler calls `activateNextPhase` → but since Phase 3 requires a gate, it returns immediately → no phase 4 tasks are seeded → brand-details and all downstream work is blocked.

The correct flow for a gate phase should be: Phase 3 completes → `completePhase` moves it to `gate-waiting` (not `completed`) → `floor:phase-complete` is NOT emitted. But looking at `completePhase`: gate phases emit `approval:needed` and move to `gate-waiting` — they do NOT emit `floor:phase-complete`. So `activateNextPhase` never gets called for phase 3 at all through that path.

The real trigger for phase 4 seeding is `approvePhaseGate(3)` or the `approval:received` event handler. But the issue says the finance-agent task completion is blocking progression — meaning phase 3 never reaches `gate-waiting` because `checkPhaseCompletion` is blocked.

**The actual root cause confirmed**: `checkPhaseCompletion` for phase 3 calls `this.phaseManager.completePhase(floorId, 3)` which requires `phase.status === 'active'`. If the PhaseManager has phase 3 in `gate-waiting` (from a prior partial run) or `pending` (never activated), `completePhase` returns `false` silently, and the `approval:needed` event is never emitted. The phase stays stuck and the owner never gets the review gate notification.

The fix needs to: (1) ensure `checkPhaseCompletion` activates the phase if it's in `pending` state before trying to complete it, and (2) add error logging when `completePhase` returns false so failures aren't silent.
