### SECTION 1: BRAND NAME PERSISTENCE PATCHES

#### 1.1 — `selectedBrand` Write Path (updateFloor → saveFloor)

**Status: CONFIRMED DEPLOYED — but with a conditional gap**

From `src/orchestrator/index.ts`, `updateFloor()`:
```typescript
if (floor.selectedBrand !== prev) {
  // ...
  saveFloor(floor).then(ok => {
    if (!ok) console.warn(`[Floor] saveFloor failed...`);
    else console.log(`[Floor] selectedBrand persisted to DB...`);
  }).catch(...);
  return floor; // early return
}
// ...
saveFloor(floor).catch(() => {}); // fallback for non-selectedBrand updates
```

**Finding:** The `selectedBrand` write path calls `saveFloor(floor)` with a `.then()` handler that logs success/failure — meaning a DB-confirmed write is attempted. However, the **early `return floor`** before the `.catch()` means if `saveFloor` rejects after the `.then()` handler throws, it's silently swallowed. The write is *attempted* correctly but success is not guaranteed to reach DB depending on Supabase connection state.

**Gap:** `saveFloor` is called as fire-and-forget in the non-brand-selection path (`saveFloor(floor).catch(() => {})`). If the Supabase write fails silently, no retry occurs and the brand name is only in-memory until the next explicit save.

#### 1.2 — `selectedBrand` written to workspace file

**Status: CONFIRMED DEPLOYED**

```typescript
this.workspace.writeFile(floor.slug, 'brand/selected-brand.md', brandSummary).catch(() => {});
```

The brand summary is written to `brand/selected-brand.md` as a secondary persistence mechanism. This file survives server restarts if the workspace directory is on persistent storage. However, this is a **file-based fallback**, not the primary DB store.

#### 1.3 — `selectedBrand` broadcast via Supabase Realtime

**Status: CONFIRMED DEPLOYED**

```typescript
broadcastFloorEvent(floorId, 'brand:selected', { floorId, brandName: sb.name, brandState: floor.brandState }).catch(() => {});
```

Frontend realtime subscription receives this event. However, if the frontend is offline or not subscribed at broadcast time, the event is lost — it is not replayed on reconnect.

#### 1.4 — Brand name recovery on server restart (`loadPersistedState`)

**Status: PARTIALLY DEPLOYED — recovery path exists but has a critical gap**

From `loadPersistedState()`:
```typescript
const hasBrand = !!(
  floor.themeConfig?.primaryColor &&
  floor.themeConfig?.headingFont &&
  floor.themeConfig.headingFont !== 'Token'
);

if (!hasBrand && floor.currentPhase >= 4) {
  // Try to read brand-visual-system.md and brand-voice-guide.md...
}
```

**Critical Finding:** The recovery block checks `themeConfig` (visual/font data), NOT `selectedBrand`. The `selectedBrand` field (which contains the brand name, tagline, personality) is loaded from Supabase via `loadFloors()` — only if the `floors` table row persisted it correctly via the prior `saveFloor()` call.

**If `saveFloor()` failed silently when brand was selected, `selectedBrand` will be `null` after restart**, and the recovery block does NOT attempt to recover `selectedBrand` from the workspace file (`brand/selected-brand.md`). The file was written but never read back during recovery.

#### 1.5 — `themeConfig` extraction and persistence

**Status: CONFIRMED DEPLOYED**

`extractBrandState()` populates `floor.themeConfig` and calls `await saveFloor(floor)` (awaited, not fire-and-forget). This is the most reliable write path in the codebase for brand data.

#### 1.6 — Brand name in `themeConfig` vs `selectedBrand`

**Status: DESIGN GAP — Not a patch failure, but an architectural risk**

The brand *name* lives in `floor.selectedBrand.name`. The `themeConfig` stores visual/voice data (colors, fonts, voice principles, tagline). These are separate fields. If only `themeConfig` recovers correctly but `selectedBrand` is lost, agents in phase 5+ will have visual brand context but lose the brand name used for copy/content generation.

### SECTION 2: CHAT HISTORY DELETION PATCHES

#### 2.1 — `clearFloorChat()` in-memory deletion

**Status: CONFIRMED DEPLOYED**

```typescript
async clearFloorChat(floorId: string): Promise<boolean> {
  this.chatHistories.delete(floorId);
  return clearChatMessages(floorId);
}
```

The in-memory map entry is deleted before the DB call. This is correct — memory is cleared synchronously regardless of DB outcome.

#### 2.2 — `clearChatMessages()` Supabase call

**Status: CONFIRMED DEPLOYED IN CODE — DB execution unverifiable from source alone**

`clearChatMessages` is imported from `src/integrations/supabase.js` and called with `return clearChatMessages(floorId)` — the return value (boolean) is propagated to the caller. The function is not in the provided source files, so its internal implementation (whether it does a hard delete, soft delete, or has RLS restrictions) cannot be verified here.

**Finding:** The route handling `clearFloorChat` is not visible in the provided `src/server/routes/chat.js` (file not included in audit). The API route must exist for the frontend to trigger deletion. If the route calls `orchestrator.clearFloorChat()` correctly and awaits the result, the patch is complete.

#### 2.3 — Chat deletion persistence across reload

**Status: CONDITIONALLY CORRECT — gap in loadChatHistory logic**

From `loadChatHistory()`:
```typescript
const dbMessages = await loadChatMessages(floorId);
if (dbMessages.length > 0) {
  this.chatHistories.set(floorId, dbMessages.map(...));
  return;
}
// Supabase returned [] — set empty cache
this.chatHistories.set(floorId, []);
```

After a successful `clearChatMessages()` DB delete, `loadChatMessages()` returns `[]`. The code then sets `this.chatHistories.set(floorId, [])` — **correctly preventing file fallback from loading stale data**. 

**However:** The file fallback path below is only reached if `loadChatMessages` **throws** (Supabase unavailable), not if it returns empty. If Supabase was unavailable during deletion but the local JSON file still has messages, those messages will reappear on restart. This is an edge case but represents a potential "ghost messages" scenario.

#### 2.4 — Chat deletion broadcast

**Status: NOT IMPLEMENTED**

There is no `broadcastFloorEvent(floorId, 'chat:cleared', ...)` call in `clearFloorChat()`. If multiple browser tabs or sessions are connected, other sessions will not receive a realtime notification that chat was cleared. They will show stale data until their next reload or until the Supabase realtime subscription detects the row deletions (if configured).

#### 2.5 — Chat history deletion does not reappear after session reload

**Status: CONFIRMED — if Supabase write succeeded**

The `loadChatHistory()` function checks `if (this.chatHistories.has(floorId)) return` first — meaning once the empty cache is set after deletion, the same session won't reload from DB. On a fresh server restart, `loadChatMessages()` will return `[]` (since rows were deleted from DB), and the file fallback is skipped because Supabase responded successfully.

#### 4.3 — Async write failure risk (race conditions)

The main race condition risk is in `updateFloor()`:

```typescript
// selectedBrand path:
saveFloor(floor).then(ok => { ... }).catch(err => { ... });
return floor; // returns immediately — caller gets stale-looking response if DB fails
```

The API response returns the floor object **before** the DB write completes. If the frontend receives a `200 OK` and assumes persistence, but the DB write fails 500ms later, the frontend state and DB state diverge. This is not a code bug introduced by a patch — it's a pre-existing architectural pattern — but it means "API returned success" does not guarantee "data persisted to DB."

### FLAGS REQUIRING OWNER REVIEW BEFORE RE-APPLICATION

**HIGH PRIORITY — Brand name recovery gap:**
The `brand/selected-brand.md` file is written on brand selection but never read back during `loadPersistedState()`. If the Supabase `saveFloor()` call silently failed at the moment of brand selection (network blip, RLS issue, connection timeout), the brand name will be lost on server restart. The workspace file exists as a fallback but the recovery code doesn't use it.

**MEDIUM PRIORITY — `saveFloor()` fire-and-forget in general path:**
The `updateFloor()` non-brand-selection path uses `saveFloor(floor).catch(() => {})` which silently swallows all errors. Any floor update (name change, budget change, etc.) that fails to persist will appear successful to the frontend but not survive a restart.

**LOW PRIORITY — Local chat file not cleared on deletion:**
When `clearFloorChat()` runs, it clears Supabase but does not delete `data/chats/{floorId}.json`. If Supabase becomes unavailable after a deletion, the file fallback will resurrect deleted messages on next load.
