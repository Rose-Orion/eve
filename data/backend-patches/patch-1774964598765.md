## Diagnosis

This is a multi-part investigation. Let me systematically analyze each of the 5 points before writing any patches.

**Key finding from the code:**

Looking at `setupEventHandlers()` in `index.ts`, the `task:completed` handler calls `checkPhaseCompletion()` which checks if all phase tasks are terminal. However, there's a critical bug: the `checkPhaseCompletion` method is called with `completedTask.phaseNumber`, but the **completed array contains 14 tasks across multiple phases**. For Phase 8 specifically, the two tasks are `ad-creative-production` and `conversion-tracking` — both listed as completed. If both are complete, Phase 8 should have triggered `completePhase()` which (since Phase 8 `requiresGate: true`) would set it to `gate-waiting`, NOT auto-advance.

**The actual stall:** Phase 8 is a gate phase (`requiresGate: true`). When both Phase 8 tasks complete, `completePhase()` fires `approval:needed` (gate-waiting), NOT `floor:phase-complete`. So `activateNextPhase()` is never called, and Phase 9 tasks are never seeded. This is **correct behavior** — Phase 8 requires owner gate approval before Phase 9 starts.

**BUT** — looking at the completed array: `ad-creative-production`, `conversion-tracking` are in it, AND so are `analytics-setup`, `launch-ad-campaign` (Phase 7), AND Phase 6 tasks. This means the floor is at or past Phase 8. The question is whether the gate was approved or is waiting.

The deeper bug I see: In `loadPersistedState`, the floor consistency check at the bottom resets `'dispatched' || 'working'` tasks to `'queued'` — but this also triggers `checkPhaseCompletion` only if tasks complete. If the server restarted while Phase 8 tasks were in `dispatched`/`working`, they'd be reset to `queued` but never re-dispatched if there's no concurrency slot or if `checkPhaseCompletion` doesn't re-evaluate.

**Most likely root cause for the reported stall:** The Phase 8 gate is in `gate-waiting` state and awaiting owner approval — OR the `task:completed` event handler for Phase 8 tasks silently failed to fire `checkPhaseCompletion` after a server restart restored tasks with `restoreWithStatus` (which doesn't trigger any events). Restored-with-status completed tasks never emit `task:completed`, so `checkPhaseCompletion` was never called for Phase 8, leaving it stuck in `active` instead of transitioning to `gate-waiting`.

**Point 4 analysis:** The completed array has all 14 tasks listed. The `checkPhaseCompletion` in `loadPersistedState` is called via `this.checkPhaseCompletion(floor.id, floor.currentPhase)` — but only for the CURRENT phase, not for all phases that have completed tasks. If `floor.currentPhase = 8` but Phase 8 tasks are restored as `completed` via `restoreWithStatus`, `checkPhaseCompletion(floor.id, 8)` would be called... but it checks `phaseManager.completePhase()` which requires the phase to be `active`. After `forceRecoveryActivate`, Phase 8 would be set to `active`, so this should work.

The real issue: `checkPhaseCompletion` is called once during recovery, but `completePhase` for a gate phase only emits `approval:needed`, NOT `floor:phase-complete`. So Phase 9 tasks are never seeded. The owner must approve the Phase 8 gate.

**For Point 5 (manual insertion):** I need to add a diagnostic/repair endpoint that can: query Phase 8 task status, verify the gate state, and if Phase 9 tasks are missing, either trigger gate approval programmatically (owner action required) or insert Phase 9 tasks directly.
