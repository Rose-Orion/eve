## Diagnosis

The `task:completed` event handler in `setupEventHandlers()` calls `checkPhaseCompletion()` after cascading dependency triggers via `dependencyGraph.onTaskCompleted()`. The critical bug is that **dependency triggers run first, then `checkPhaseCompletion` is called — but `checkPhaseCompletion` only evaluates the completing task's phase, not the newly-ready tasks' phases**. More importantly, when `activateNextPhase()` calls `seedNextPhaseTasks()`, the newly-created tasks are registered in the dependency graph **after** the event handler returns, so the `dependencyGraph.onTaskCompleted()` cascade at the top of the handler cannot see them. Additionally, there is no structured logging at the handler entry point to confirm the event fired and dependency config was evaluated — making silent failures invisible.

The second issue: `checkPhaseCompletion` has no logging for *why* it returns early (empty phase tasks, not-all-done, gate-block), so when it silently returns, there is no trace of what was evaluated.

## Verification

1. **Confirm event fires**: After deploying, complete any task on Quest Kids Phase 1 or SideQuest Phase 6. You should see `[EventHandler:task:completed] Event fired —` in logs with the correct taskType and phaseNumber.

2. **Confirm dependency evaluation**: If the task has dependents, you'll see `[EventHandler:task:completed] Dependency cascade for task ...` listing how many tasks were unblocked.

3. **Confirm phase check runs**: You'll see `[PhaseCompletion] Evaluating phase N for "Floor Name": N task(s) found (taskType=status, ...)`.

4. **Confirm gate or advance decision**: Either `[activateNextPhase] Phase N requires gate approval` (gate phases 3, 6, 8) or `[activateNextPhase] Auto-advancing` for non-gate phases.

5. **Quest Kids Phase 1 specifically**: Phase 3 is a gate phase — after all tasks complete you should see `gate-waiting` status and `[activateNextPhase] Phase 3 requires gate approval`. If instead you see `phaseManager.completePhase() returned false` with `status="gate-waiting"`, that means the gate already fired correctly on a prior task completion and the second call is idempotent.

6. **SideQuest Phase 6 specifically**: Phase 6 is also a gate phase — same pattern applies.
