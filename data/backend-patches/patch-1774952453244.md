## Diagnosis

This is a **deployment persistence and handler registration problem**, not a simple bug. The `budget-plan → brand-details` transition handler isn't a single isolated event listener — it's driven by the `task:completed` event in `setupEventHandlers()` calling `checkPhaseCompletion()`, which calls `phaseManager.completePhase()`, which emits `floor:phase-complete`, which calls `activateNextPhase()`, which calls `seedNextPhaseTasks()`. The `seedNextPhaseTasks()` for phase 3 (Foundation Sprint) does NOT include a `brand-details` task — it seeds `brand-options`, `business-strategy`, and `budget-plan`. The brand visual and voice work happens in **Phase 4** (`brand-visual-system`, `brand-voice-guide`). There is no task named `brand-details` anywhere in the codebase.

The root cause is therefore: **the transition from budget-plan completion (Phase 3) to Phase 4 tasks never fires** because Phase 3 has a gate (`requiresGate: true`), meaning `completePhase()` transitions to `gate-waiting` instead of `floor:phase-complete`. Phase 4 only seeds when the owner explicitly calls `approvePhaseGate(3)` or `approveFoundation()`. Additionally, the Phase 3 completion is **blocked** by the superseded budget-plan gate in `checkPhaseCompletion()` — if any budget-plan task is marked as superseded (reviewStatus='rejected'), the phase won't complete even after the revised task finishes, unless the `validCompletedBudgetPlan` check passes.

The 10 prior patches failed to persist because they were patching runtime state (requeuing tasks, emitting events) rather than fixing the **structural gate**: Phase 3 requires owner gate approval before Phase 4 tasks are seeded. No amount of task requeuing will trigger Phase 4 without that gate being approved.

**Root cause documentation for future reference**: The `budget-plan → brand-details` framing is a misnomer. Phase 3 completes when all tasks (brand-options, business-strategy, budget-plan) are terminal. It then transitions to `gate-waiting` (not `phase-complete`) because Phase 3 has `requiresGate: true`. The fix is: surface this clearly to the owner, not patch the code further.
