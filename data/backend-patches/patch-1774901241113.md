## Diagnosis

The issue has two parts: (1) completed budget-plan tasks with the $200 baseline need their `reviewStatus` set to `'rejected'` and `reviewFeedback` stamped with the supersession note in the database, and (2) Phase 1 (Foundation Sprint = phase 3) must not be allowed to mark complete while the old $200 task is the only completed budget-plan task — the phase completion check needs to block on this condition until the new $500 task is approved/completed.

The `loadAllTasks` function in `supabase.ts` currently loads `reviewStatus` and `reviewFeedback` as always `'pending'`/`null`, so stale superseded tasks restored from DB lose their invalidation markers. The `checkPhaseCompletion` in `orchestrator/index.ts` does not inspect budget-plan staleness before allowing phase 3 to complete.

```typescript
  private checkPhaseCompletion(floorId: string, phaseNumber: number): void {
    const floorTasks = this.taskManager.getFloorTasks(floorId);
    const phaseTasks = floorTasks.filter(t => t.phaseNumber === phaseNumber);
    if (phaseTasks.length === 0) return;

    // Treat escalated as terminal (failed permanently) — phase can still complete
    const TERMINAL_STATUSES: string[] = ['completed', 'escalated'];
    const allDone = phaseTasks.every(t => TERMINAL_STATUSES.includes(t.status));
    if (!allDone) return;

    // GATE: For phase 3 (Foundation Sprint), block completion if the only completed
    // budget-plan task is the stale $200 one (marked superseded via reviewStatus='rejected'
    // with the supersession note). Phase 3 must not complete until either:
    //   (a) a valid $500-baseline budget-plan task is completed and NOT superseded, OR
    //   (b) there are no budget-plan tasks at all in this phase (non-blocking path).
    const SUPERSESSION_NOTE =
      'Revised budget from $200 to $500 — all prior budget-plan outputs are superseded ' +
      'and should not be used for downstream planning.';

    if (phaseNumber === 3) {
      const budgetPlanTasks = phaseTasks.filter(t => t.taskType === 'budget-plan');
      if (budgetPlanTasks.length > 0) {
        const validCompletedBudgetPlan = budgetPlanTasks.some(
          t =>
            t.status === 'completed' &&
            !(t.reviewStatus === 'rejected' && t.reviewFeedback === SUPERSESSION_NOTE),
        );
        if (!validCompletedBudgetPlan) {
          // All completed budget-plan tasks are superseded (stale $200 outputs).
          // Phase 3 is blocked until the new $500 task completes.
          console.log(
            `[Floor] Phase 3 completion BLOCKED for ${floorId}: ` +
            `all completed budget-plan tasks are superseded ($200 baseline). ` +
            `Waiting for $500 budget-plan task to complete.`,
          );
          return;
        }
      }
    }

    const completed = this.phaseManager.completePhase(floorId, phaseNumber);
    if (completed) {
      console.log(`[Floor] Phase ${phaseNumber} complete for ${floorId}`);
      // Floor:phase-complete event triggers activateNextPhase (for non-gate phases)
    }
  }
```

## Verification

1. **Stale task accessibility**: After server restart, load the Quest Kids floor and call `GET /api/floors/:id/tasks`. The old $200 budget-plan task should show `reviewStatus: "rejected"` and `reviewFeedback` containing the supersession note. This confirms it is not accessible as a valid completed output.

2. **Phase 3 gate block**: With only the stale task in DB, trigger `checkPhaseCompletion` (by marking a non-budget-plan task as escalated/completed to fill out the phase). Check that Phase 3 does **not** advance to `gate-waiting`. Logs should show: `Phase 3 completion BLOCKED for ... all completed budget-plan tasks are superseded`.

3. **Phase 3 unblocks correctly**: Once the new $500 budget-plan task (`prompt` contains `'Revised budget from $200 to $500'`, `reviewStatus !== 'rejected'`) completes, `checkPhaseCompletion` should allow Phase 3 to proceed to `gate-waiting` normally.

4. **DB roundtrip**: Verify via Supabase dashboard that the old task row has `review_status = 'rejected'` and `review_feedback` = supersession note text (this is set by `invalidateAndRequeueBudgetPlan` → `markBudgetPlanTasksSuperseded` → `saveTask`).

## Rollback

**Change 1 & 3** (supabase.ts): Revert the two `reviewStatus`/`reviewFeedback` lines back to hardcoded `'pending'` and `null`. The only functional regression is that supersession markers won't survive server restarts — the invalidation still works in-memory within a session.

**Change 2** (orchestrator/index.ts): Remove the `if (phaseNumber === 3)` block from `checkPhaseCompletion` (lines between the `// GATE:` comment and `const completed = ...`). Phase 3 will then complete as before, regardless of whether the budget-plan task is superseded. This is the safe no-op rollback — it removes the protection but causes no crashes.

No database migrations required. No environment variables touched.
