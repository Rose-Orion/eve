# EVE — Website Build Workflow
## How Agents Build a Production Ecommerce Site

---

# KEY RESEARCH FINDING

Next.js 16.2 (March 2026) was specifically designed for AI agent development. The Next.js team found that giving agents access to bundled documentation achieved a **100% pass rate** on Next.js evals — vs 79% for skill-based approaches. Their recommendation: always have agents read `node_modules/next/dist/docs/` before writing any code.

**What this means for EVE:** The Web Agent's workspace must include the AGENTS.md file from Next.js and the CLAUDE.md file that references it. This single change will dramatically improve the quality of generated code.

---

# THE TECH STACK (CONFIRMED)

| Layer | Technology | Why |
|---|---|---|
| Framework | Next.js 16 (App Router) | SSR, RSC, Server Actions, streaming, edge functions, agent-friendly |
| Language | TypeScript | Type safety catches errors AI agents commonly make |
| Styling | Tailwind CSS | Utility-first, easy for agents to write, no CSS files to manage |
| Payments | Stripe | Industry standard, best docs, strongest API |
| Database | Supabase (PostgreSQL) | Real-time, auth built in, good free tier, edge functions |
| Hosting | Vercel | Native Next.js support, preview deployments, edge network |
| Email (transactional) | Resend | Simple API, good deliverability, developer-friendly |
| Email (marketing) | ConvertKit | Creator-focused, automation flows, good API |
| Analytics | GA4 + Meta Pixel | Standard tracking for ads attribution |
| Image CDN | Vercel Image Optimization | Built into Next.js, automatic WebP/AVIF, responsive sizing |

# THE BUILD SEQUENCE

## Stage 1: Project Scaffolding (Web Agent, 10 min)

The Web Agent creates the project using Next.js's built-in scaffolding, then configures it for AI-assisted development.

```bash
# Web Agent executes:
cd __PATH_EVE_PROJ__{floorName}/website
npx create-next-app@latest . --typescript --tailwind --eslint --app --src-dir
```

Then immediately sets up agent-friendly development:

```bash
# Generate AGENTS.md for AI-assisted development
npx @next/codemod agents

# This creates:
# - AGENTS.md (tells agents to read bundled docs)
# - CLAUDE.md (Claude Code integration)
# - .next/docs/ reference available
```

## Stage 2: Design System Setup (Design Agent → Web Agent, 2-4 hours)

**Design Agent produces wireframes and visual specs.** These go to the shared workspace as design files + a design tokens file.

**Design tokens file** (created by Design Agent, consumed by Web Agent):

```typescript
// src/lib/design-tokens.ts
// Generated by Design Agent from Foundation Package

export const tokens = {
  colors: {
    primary: '#1A1A2E',      // From Foundation Package
    secondary: '#E94560',
    accent: '#F5C518',
    background: '#FAFAFA',
    surface: '#FFFFFF',
    text: '#1A1A2E',
    textMuted: '#6B7280',
    success: '#10B981',
    error: '#EF4444',
  },
  fonts: {
    heading: '"Oswald", sans-serif',
    body: '"Inter", sans-serif',
    mono: '"JetBrains Mono", monospace',
  },
  spacing: {
    section: '80px',
    container: '1280px',
  },
  borderRadius: {
    sm: '4px',
    md: '8px',
    lg: '16px',

  },

```

**Web Agent configures Tailwind** with the brand's design tokens:

```typescript
// tailwind.config.ts
// Web Agent extends Tailwind with brand tokens

import type { Config } from 'tailwindcss'

const config: Config = {
  content: ['./src/**/*.{ts,tsx}'],
  theme: {
    extend: {

        primary: '#1A1A2E',

        heading: ['"Oswald"', 'sans-serif'],
        body: ['"Inter"', 'sans-serif'],
      },
    },
  },
  plugins: [],
}
export default config
```

## Stage 3: Component Library (Web Agent, 4-8 hours)

Web Agent builds the reusable component library based on Design Agent's wireframes. These are the building blocks every page uses.

**Build order (dependency-driven):**

```
1. Layout components (Header, Footer, Container, MobileNav)
   └── No dependencies — build first
   
2. UI primitives (Button, Input, Badge, Card, Modal, Toast)
   └── No dependencies — build alongside layout

3. Product components (ProductCard, ProductGallery, ProductGrid, QuickView)
   └── Depends on: UI primitives + design tokens

4. Cart components (CartDrawer, CartItem, CartSummary)
   └── Depends on: UI primitives + product types

5. Checkout components (CheckoutForm, OrderSummary, AddressForm)
   └── Depends on: UI primitives + Stripe integration

6. Marketing components (Hero, FeaturedProducts, Newsletter, Testimonials)
   └── Depends on: UI primitives + design tokens

7. Content components (FAQ Accordion, PolicyPage, AboutSection)
   └── Depends on: UI primitives
```

**Quality rules for Web Agent:**

```
EVERY COMPONENT MUST:
- Be mobile-first (375px → scales up)
- Use TypeScript with strict types (no `any`)
- Use Tailwind classes only (no inline styles, no CSS modules)
- Be server components by default (add 'use client' only when needed)
- Handle loading states (skeleton screens, not spinners)
- Handle empty states (not blank screens)
- Handle error states (graceful fallback, not crash)
- Be accessible (semantic HTML, aria labels, keyboard nav)
- Use Next.js Image component for all images (automatic optimization)
- Use the brand's design tokens for all colors, fonts, spacing
```

## Stage 4: Page Assembly (Web Agent + Copy Agent, 4-8 hours)

**Copy Agent delivers page content** to the workspace. Web Agent integrates it into the pages.

**Homepage build (the most important page):**

```
ABOVE THE FOLD (first screen, must convert):
  ├── Hero section
  │   ├── Headline (from Copy Agent — value proposition)
  │   ├── Subheadline (from Copy Agent — supporting statement)
  │   ├── CTA button (from Copy Agent — "Shop Now" or similar)
  │   └── Hero image (from Design Agent — generated with brand style)
  │
  ├── Social proof strip (if available — review count, trust badges)
  │
BELOW THE FOLD (scroll reveals):
  ├── Featured products grid (from Commerce Agent catalog)
  ├── Brand story section (from Copy Agent about text + Design Agent imagery)
  ├── Category showcase (product categories with images)
  ├── Testimonials (placeholder or from real reviews)
  ├── Newsletter signup (email capture + Copy Agent CTA)
  └── Footer (links, policies, social icons)
```

**Product page build:**

```
PRODUCT PAGE STRUCTURE:
  ├── Product gallery (multiple images, swipe on mobile, zoom on desktop)
  ├── Product info
  │   ├── Name + price (from Commerce Agent)
  │   ├── Description (from Copy Agent)
  │   ├── Variant selector (size, color — from Commerce Agent)
  │   ├── Add to cart button
  │   ├── Stock indicator
  │   └── Shipping estimate
  ├── Product details accordion (materials, care, sizing)
  ├── Related products grid
  └── Reviews section (placeholder for post-launch)
```

**Page build order:**

```
1. Homepage (most important — sets the tone)
2. Product listing page (grid + filters)
3. Product detail page (the conversion page)
4. Cart page/drawer
5. Checkout flow (Stripe integration)
6. Order confirmation / success page
7. About page
8. FAQ page
9. Privacy policy + Terms of service
10. 404 page
```

**Legal Pages Note:**
Pages 9 (Privacy Policy, Terms of Service, Return/Refund Policy, Cookie Policy) are generated by the Copy Agent using established ecommerce legal templates customized with the floor's business details (company name, contact email, shipping timeframe, return window). These are functional drafts, not legal advice. The Launch Agent QA checklist flags them with: "Legal pages generated from templates. Recommend owner review with a lawyer before handling significant transaction volume." For v1 test floors with low volume, the template-generated versions are sufficient to launch.

## Stage 5: Ecommerce Integration (Web Agent + Commerce Agent, 4-8 hours)

**Product data flow:**

```
Commerce Agent → product catalog (JSON) → Supabase database
                                        → Web Agent reads via API
                                        → Product pages render from database
```

**Stripe integration:**

```
CHECKOUT FLOW:
1. Customer adds products to cart (client-side state)
2. Customer clicks "Checkout" → Stripe Checkout Session created (server action)
3. Customer redirected to Stripe's hosted checkout (PCI compliant — we never touch cards)
4. Payment succeeds → Stripe webhook fires → /api/stripe/webhook/route.ts
5. Webhook handler:
   a. Verify webhook signature (security)
   b. Create order record in Supabase
   c. Send order confirmation email via Resend
   d. If POD: forward order to fulfillment provider API
   e. Update analytics (conversion event)
6. Customer redirected to /checkout/success with order details
```

**Email capture integration:**

```
NEWSLETTER FLOW:
1. Customer enters email → /api/email/subscribe/route.ts
2. API route:
   a. Validate email
   b. Add to ConvertKit via API (tagged as "website signup")
   c. Trigger welcome sequence in ConvertKit
   d. Return success/error to frontend
3. Frontend shows confirmation toast
```

## Stage 6: Analytics & Tracking (Web Agent + Analytics Agent, 2-4 hours)

```
TRACKING SETUP:
  ├── GA4 (Google Analytics 4)
  │   ├── Pageview tracking (automatic with Next.js)
  │   ├── E-commerce events: view_item, add_to_cart, begin_checkout, purchase
  │   ├── Custom events: newsletter_signup, product_filter_used
  │   └── Implemented via gtag.js in root layout
  │
  ├── Meta Pixel
  │   ├── PageView (all pages)
  │   ├── ViewContent (product pages)
  │   ├── AddToCart
  │   ├── InitiateCheckout
  │   ├── Purchase (with value and currency)
  │   └── Implemented via pixel script in root layout
  │
  ├── UTM Parameters
  │   ├── Analytics Agent defines UTM structure for all marketing links
  │   ├── utm_source, utm_medium, utm_campaign, utm_content
  │   └── Tracked through to purchase for attribution
  │
  └── Cookie Consent
      ├── Banner component (Web Agent implements)
      ├── GA4 and Meta Pixel only fire after consent
      ├── Essential cookies (cart, session) don't require consent
      └── Consent choice persisted in localStorage
```

## Stage 7: Performance Optimization (Web Agent, 2-4 hours)

```
PERFORMANCE CHECKLIST:
  ├── Images
  │   ├── All images use Next.js Image component (automatic WebP/AVIF)
  │   ├── Lazy loading for below-fold images
  │   ├── Proper width/height to prevent layout shift
  │   ├── Priority loading for hero image and first product images
  │   └── Placeholder blur for loading state
  │
  ├── Code
  │   ├── Server Components by default (minimal client JS)
  │   ├── Dynamic imports for heavy components (cart drawer, modals)
  │   ├── Route-based code splitting (automatic with App Router)
  │   └── No unused dependencies
  │
  ├── Data
  │   ├── Server-side data fetching (no client-side API calls for initial load)
  │   ├── Incremental Static Regeneration for product pages
  │   ├── Edge caching for static assets
  │   └── Streaming with Suspense for slow data
  │
  ├── SEO + GEO
  │   ├── Dynamic metadata per page (title, description, OG tags)
  │   ├── JSON-LD structured data (Product, Organization, BreadcrumbList)
  │   ├── Sitemap generated at build time
  │   ├── robots.txt configured
  │   ├── Canonical URLs set
  │   └── AI-parseable product data for GEO
  │
  └── Core Web Vitals targets
      ├── LCP (Largest Contentful Paint): < 2.5s
      ├── FID (First Input Delay): < 100ms
      ├── CLS (Cumulative Layout Shift): < 0.1
      └── TTFB (Time to First Byte): < 200ms (Vercel edge)
```

## Stage 8: Mobile Optimization (Web Agent, 2-4 hours)

**70%+ of ecommerce traffic is mobile. Mobile is the primary design target.**

```
MOBILE CHECKLIST:
  ├── Touch targets: minimum 44x44px for all interactive elements
  ├── Font sizes: body 16px minimum (prevents iOS zoom on input focus)
  ├── Navigation: hamburger menu or bottom nav (not horizontal scroll)
  ├── Product gallery: swipe gestures, full-width images
  ├── Cart: slide-out drawer (not full page redirect)
  ├── Checkout: single column, large input fields, mobile keyboard types
  ├── CTAs: sticky "Add to Cart" button at bottom of product page
  ├── Search: full-screen search overlay on mobile
  └── Testing: tested at 375px (iPhone SE), 390px (iPhone 14), 414px (iPhone Plus)
```

# THE HANDOFF CHAIN

## Who Produces What → Who Consumes It

```
FOUNDATION PACKAGE (approved)
  │
  ├──→ Design Agent reads: visual direction, colors, typography, mood
  │    PRODUCES: wireframes, mockups, design tokens file
  │    DELIVERS TO: /design/ workspace folder
  │
  ├──→ Copy Agent reads: brand voice, target customer, product strategy
  │    PRODUCES: all page copy, product descriptions, CTAs
  │    DELIVERS TO: /copy/ workspace folder
  │
  ├──→ Commerce Agent reads: product strategy, pricing
  │    PRODUCES: product catalog JSON, pricing, variant data
  │    DELIVERS TO: /product/ workspace folder
  │
  └──→ Web Agent reads: ALL of the above
       CONSUMES FROM: /design/, /copy/, /product/
       PRODUCES: working Next.js website
       DELIVERS TO: /website/ workspace folder (the actual codebase)

DEPENDENCY FLOW:
  Design Agent → design tokens + wireframes → Web Agent starts building
  Copy Agent → page content → Web Agent integrates (can work in parallel)
  Commerce Agent → product data → Web Agent integrates (can work in parallel)
  Design Agent → product images → Web Agent adds to site
  
  Web Agent doesn't wait for everything — it builds the structure first,
  then integrates content as it arrives from other agents.
```

# PREVIEW + DEPLOYMENT PIPELINE

```
LOCAL PREVIEW (Phases 3-5):
  Web Agent runs: npm run dev
  Preview at: localhost:3001 (port assigned per floor)
  Phone testing: QR code with local IP (http://192.168.x.x:3001)
  Hot reload: changes appear instantly in browser
  
  You review on your phone. Chat with Floor Manager for feedback.
  Web Agent fixes. Preview updates. Loop until satisfied.

STAGING (Phase 6):
  Web Agent deploys: vercel deploy (preview, not production)
  Preview URL: faithforge-preview-abc123.vercel.app
  Shareable, on the internet, but not indexed by Google
  You test checkout with Stripe test cards
  Launch Agent runs QA checklist against staging URL

PRODUCTION (Phase 8, after Gate 2 approval):
  Web Agent deploys: vercel deploy --prod
  Production URL: faithforge.vercel.app (or custom domain)
  Stripe switches to live mode
  Analytics verified on production URL
  DNS configured if custom domain approved
```

# ERROR RECOVERY

```
BUILD FAILS:
  Web Agent reads the error output (Next.js 16 forwards all errors to terminal)
  Diagnoses: syntax error? type error? missing dependency? config issue?
  Fixes and rebuilds
  If can't fix after 3 attempts → Floor Manager investigates

DEPLOYMENT FAILS:
  Check Vercel error log
  Common causes: build error (fix code), env var missing (check .env), 
  rate limit (wait and retry)
  Web Agent fixes and redeploys
  Persistent failure → Floor Manager notifies you with error details

STRIPE WEBHOOK FAILS:
  Retry queue processes missed webhooks when connectivity restores
  All order state in database — no data lost even if webhook delayed
  Floor Manager alerts if webhooks failing for 30+ minutes

SITE CRASHES POST-LAUNCH:
  Next.js error boundary catches client errors (shows fallback UI, not blank page)
  Server errors logged and reported to Floor Manager
  Web Agent hotfixes critical issues
  Vercel instant rollback available if new deploy breaks the site
```

# QUALITY GATES FOR WEBSITE

Before the site moves from each stage to the next, these checks must pass:

```
SCAFFOLDING → COMPONENTS:
  □ Project builds without errors
  □ TypeScript strict mode enabled
  □ Tailwind configured with brand tokens
  □ AGENTS.md present for AI-assisted development

COMPONENTS → PAGES:
  □ All components render correctly
  □ Mobile responsive (375px tested)
  □ Loading, empty, and error states handled
  □ Accessibility basics (semantic HTML, keyboard nav)

PAGES → INTEGRATION:
  □ All pages load with real content
  □ Navigation works (all links, back button)
  □ Products display from database
  □ Brand Agent approves: visual consistency, brand match

INTEGRATION → STAGING:
  □ Stripe test checkout completes successfully
  □ Order confirmation email sends
  □ Analytics events fire correctly
  □ Email capture works
  □ Cookie consent functional

STAGING → PRODUCTION (Gate 2):
  □ Full Launch Agent QA checklist passes
  □ Core Web Vitals within targets
  □ SSL active
  □ Privacy policy and terms live
  □ Mobile tested on 3 screen sizes
  □ Owner approval received

PRODUCTION → LIVE:
  □ Stripe live mode verified
  □ Webhook signatures verified
  □ Analytics confirmed on production domain
  □ First test order placed and fulfilled
  □ DNS propagated (if custom domain)
```

# POST-LAUNCH: WEBSITE A/B TESTING

Once the site is live and has traffic, the Web Agent and Analytics Agent can run conversion optimization tests:

```
TESTABLE ELEMENTS:
  - Hero headline (value proposition wording)
  - CTA button text and color
  - Product page layout (image left vs. image top)
  - Free shipping threshold ($50 vs $60 vs $75)
  - Checkout flow (1-page vs multi-step)
  - Social proof placement (above fold vs below)

IMPLEMENTATION:
  Simple cookie-based split: 50/50 traffic allocation.
  Track conversion rate per variant via GA4 custom events.
  Minimum 200 conversions per variant before declaring a winner.
  Analytics Agent monitors and reports results.
  Strategy Agent proposes tests. You approve before launch.

NOT for v1 build — add after floor has 500+ monthly visitors.
```
