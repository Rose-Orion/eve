# EVE — PromptBuilder Specification
## The Engine That Makes Every Agent Smart

---

# WHY THIS IS THE MOST IMPORTANT PIECE

Every agent in EVE is ultimately just an API call to Anthropic with a system prompt. The quality of that system prompt determines whether the agent produces professional-grade work or mediocre output. The PromptBuilder is the code that assembles that system prompt. Get this right and every agent is excellent. Get it wrong and no amount of orchestration will save you.

Research finding: "38.5% of AI conversations involve iterative refinement — the first try failed." The difference between 4.2 attempts and 1.3 attempts to get usable output is prompt structure, not model choice. EVE's agents need to hit on the first try because they're running autonomously — there's no human typing "make it better."

---

# DESIGN PRINCIPLES

## 1. Treat Prompts as System Interfaces, Not Creative Writing

The 2026 best practice: system prompts should read like API contracts, not instructions to a human. Clear inputs, clear outputs, clear constraints. Every word carries constraint or context — if it's decoration, delete it.

## 2. Structured Output Over Free Text

Force agents to output structured data (JSON) for anything that feeds into another agent or system. Free text only for creative work (copy, descriptions, scripts). This reduces errors from 38.5% to 12.3% based on production data.

## 3. Separation of Concerns

Never mix instructions with data in the prompt. Use XML-like delimiters to create clear boundaries:
- `<role>` — who this agent is
- `<brand>` — the brand context
- `<skills>` — knowledge from SKILL.md files
- `<task>` — the current assignment
- `<rules>` — constraints and permissions
- `<workspace>` — what outputs exist
- `<examples>` — few-shot examples when needed

## 4. Claude-Specific Optimization

Claude is highly sensitive to instruction clarity and follows rules literally. Straightforward, well-scoped instructions consistently outperform complex prompt constructions. State what TO do, not what NOT to do (Claude handles positive instructions better).

## 5. Token Budget Discipline

Every token costs money. The PromptBuilder must track token usage per section and stay under 8,000 tokens total. Priority order when approaching limits is defined and enforced.

# THE PROMPT TEMPLATE STRUCTURE

Every agent’s system prompt follows this exact structure. The PromptBuilder fills in each section from the appropriate data source. All sections use XML tags — research shows Claude parses XML 23% more accurately than Markdown, cutting hallucination by ~40%.

**NEW IN v2:** Voice Sample (loaded for content agents), Gold Standard Examples (approved outputs as few-shot), Boundaries (explicit scope limits), and Generated Knowledge instructions (for analysis agents).

```xml
<system>

<role>
  [AGENT IDENTITY — who you are, what you do, what you don't do]
  [MODEL TIER — Opus/Sonnet/Haiku and what that means for your behavior]
</role>

<brand_context>
  [BRAND CONTEXT — extracted from Foundation Package, tailored to this agent type]
  [Only the sections relevant to this agent's work]
  <voice_sample>
    [500-WORD BRAND VOICE REFERENCE - generated by Brand Agent, approved by owner]
    [Loaded for: Copy, Social, Ads, email tasks. NOT loaded for technical tasks]
    [~250 tokens. Eliminates tone drift - the #1 revision cause for copy.]
  </voice_sample>
</brand_context>

<expertise>
  [DOMAIN KNOWLEDGE — extracted from assigned SKILL.md files]
  [For analysis agents: includes Generated Knowledge pattern instructions]
</expertise>

<examples>
  [GOLD STANDARD OUTPUTS - 2-3 approved examples for this task type]
  [Month 1: template defaults. Month 2+: real approved outputs]
  [Max 1,000 tokens for examples section]
</examples>

<task>
  [CURRENT ASSIGNMENT — specific task from Floor Manager]
  [Acceptance criteria — what "done" looks like]
  [Dependencies — what inputs are available, what's still pending]
  [Output format — exactly what to produce and where to put it]
</task>

<workspace>
  [WORKSPACE STATE — what files/outputs already exist]
  [What other agents have produced that's relevant]
  [What's pending from other agents]
</workspace>

<rules>
  [TERMINAL ACCESS — which tier this agent has]
  [COLLABORATION — who to work with, who to escalate to]
  [QUALITY — review requirements before output is final]
  [BUDGET — cost awareness, efficiency expectations]
  [SAFETY — what this agent must never do]
</rules>

<output_format>
  [STRUCTURED OUTPUT — JSON schema for status updates, deliverable metadata]
  [When to use structured output vs. free text]
</output_format>

<boundaries>
  [WHAT THIS AGENT DOES NOT DO - explicit scope limits]
  [Prevents competence creep - agents drifting outside their lane]
  [Example: "You do NOT write code. Route to Web Agent."]
</boundaries>

</system>
```

# SECTION-BY-SECTION SPECIFICATION

## Section 1: Role

The role section defines WHO the agent is. This is loaded once per agent type and rarely changes.

**Structure:**
```xml
<role>
You are the [ROLE NAME] for [FLOOR NAME].

IDENTITY:
[One sentence: what this agent IS]

RESPONSIBILITIES:
[3-7 bullet points: specific things this agent does]

BOUNDARIES:
[2-4 bullet points: what this agent does NOT do]

MODEL TIER: [Opus/Sonnet/Haiku]
[One sentence: what this means for behavior]
- Opus: You make strategic decisions. Take time to reason deeply. Quality over speed.
- Sonnet: You execute with high quality. Balance thoroughness with efficiency.
- Haiku: You process data and execute mechanical tasks. Be fast and precise.

COMMUNICATION:
[How this agent communicates — tone, format, length]
- Status updates: structured JSON
- Creative output: free text with brand voice
- Escalations: structured with evidence and recommendation
</role>
```

**Example — Copy Agent:**
```xml
<role>
You are the Copy Agent for CaseForge.

IDENTITY:
You are the writer. Every word the business publishes comes from you.

RESPONSIBILITIES:
- Write all website copy (homepage, product pages, about, FAQ, policies)
- Write all email copy (templates, sequences, automation flows)
- Write all ad copy (headlines, body, CTAs for Meta, Google, TikTok)
- Write all social captions, hooks, hashtags, and text overlays
- Write video scripts with hook → story → CTA structure
- Write support content (knowledge base, chatbot flows, response templates)
- Adapt voice per platform (TikTok casual, Instagram aspirational, Facebook conversational)

BOUNDARIES:
- You do NOT design visuals or generate images
- You do NOT write code or implement anything on the website
- You do NOT make strategic decisions about what to sell or who to target
- You do NOT approve your own work — Brand Agent reviews all copy

MODEL TIER: Sonnet
You execute with high quality. Write clean, human-sounding copy efficiently. When unsure about brand voice, check the brand guidelines before guessing.

COMMUNICATION:
- Deliver copy as plain text files in the workspace
- Status updates as JSON: {"status": "complete", "output": "/copy/homepage.md", "needs_review": true}
- If blocked, report: {"status": "blocked", "reason": "...", "waiting_on": "agent_name"}
</role>
```

## Section 2: Brand Context

Extracted from the Foundation Package. **Crucially, each agent type gets only the sections relevant to their work.** The Copy Agent gets voice guidelines and target customer. The Design Agent gets visual direction and color palette. The Web Agent gets technical requirements and brand fonts.

**Extraction rules per agent type:**

| Agent | Gets from Foundation Package |
|---|---|
| Floor Manager | Full Foundation Package (needs complete context) |
| Brand Agent | Full Foundation Package (is the brand guardian) |
| Strategy Agent | Business model, competitive landscape, KPIs |
| Finance Agent | Financial projections, pricing strategy, budget |
| Copy Agent | Brand voice, target customer, product strategy, key differentiators |
| Web Agent | Typography, color palette (hex codes), visual style, technical requirements |
| Design Agent | Visual style, color palette, typography, mood, photography style, target customer |
| Video Agent | Visual style, mood, target customer, brand voice (for scripts), content strategy |
| Commerce Agent | Product strategy, pricing strategy, competitive landscape, target customer |
| Social Media Agent | Target customer, brand voice, content strategy, platform priorities |
| Ads Agent | Target customer, competitive landscape, pricing strategy, platform priorities, KPIs |
| Analytics Agent | KPIs, revenue targets, conversion targets |
| Launch Agent | Full Foundation Package (needs to verify everything) |

**Token budget:** 1,000-1,500 tokens for brand context. Compress by extracting only the relevant sections and summarizing where possible.

### Phased Loading: Brand Context Lifecycle

The brand context section doesn't exist at floor creation. It's built through a defined lifecycle:

**State 1: PRE-FOUNDATION (floor just created, no brand yet)**
```
Brand context = EMPTY

Only 3 agents are running: Brand Agent, Strategy Agent, Finance Agent
Their prompts use a CREATION task instead of brand context:

<brand>
  NO BRAND EXISTS YET. Your job is to CREATE IT.
  
  OWNER INPUT:
  [Answers from CEO Mode's clarifying questions]
  [Goal description, target market, preferences expressed]
  
  DELIVERABLE:
  Create 3 distinct brand direction options. For each option:
  - Brand name and tagline
  - Color palette (primary, secondary, accent with hex codes)
  - Typography (heading + body font)
  - Voice and tone (with 3 example sentences)
  - Visual mood (described in 2-3 sentences)
  - Target customer profile
  
  Present all 3 options for human owner review.
</brand>
```

**State 2: FOUNDATION REVIEW (Foundation Package created, waiting for Gate 1)**
```
Brand context = DRAFT (not yet approved)

Brand Agent + Strategy Agent + Finance Agent have produced the Foundation Package.
Floor Manager has compiled it and presented it to the human owner.
Human owner is reviewing — may approve, request changes, or pick from options.

No other agents run during this state. The floor is paused waiting for approval.
```

**State 3: FOUNDATION APPROVED (Gate 1 passed — brand is live)**
```
Brand context = ACTIVE

The approved Foundation Package is saved to:
  /brand/foundation-package.md

PromptBuilder now extracts brand context for every agent:
  - Reads /brand/foundation-package.md
  - Extracts sections per the agent-type extraction rules above
  - Injects into every agent's <brand> section

All agents can now start working. Every agent reads the same brand source.
This is the state for the remainder of the floor's lifetime.
```

**State 4: BRAND REVISION (human requests brand changes after approval)**
```
Brand context = UPDATING

This is the most disruptive change possible. When the human owner requests 
a brand change (different colors, different voice, different name):

1. Floor Manager warns: "Brand change affects every agent and every deliverable.
   Estimated impact: 1-2 days of cascading updates."
2. Human confirms → Brand Agent creates the revision
3. Human approves the revision
4. PromptBuilder reloads EVERY agent's brand context from the updated Foundation Package
5. All agents with in-progress work review their outputs against the new brand
6. Affected deliverables are flagged for re-review

The PromptBuilder tracks a version number on the Foundation Package.
If an agent's last loaded brand version < current version, it gets a 
flag in its next prompt: "BRAND UPDATED — review your current work 
against the new guidelines before continuing."
```

**Key rule:** No agent except Brand Agent, Strategy Agent, and Finance Agent ever runs with an empty brand context. The PromptBuilder enforces this — if an agent is dispatched and the Foundation Package hasn't been approved yet, the PromptBuilder blocks the API call and returns an error to the Orchestrator.

## Section 3: Expertise (replaces Skills)

Loaded from assigned SKILL.md files. Each agent gets 1-3 skills. The PromptBuilder reads the SKILL.md file, extracts the core knowledge, and compresses it to fit the token budget.

**Extraction process:**
1. Read the SKILL.md file
2. Extract the "what to do" and "how to do it" sections
3. Strip examples that aren't directly relevant to the current task
4. Compress to 800-2,000 tokens per skill
5. If total skill tokens exceed 4,000, prioritize skills most relevant to the current task

**Token budget:** 1,000-3,000 tokens per skill, max 3 skills = max 4,000-6,000 tokens. In practice, aim for 3,000 total across all skills by compressing aggressively.

## Section 4: Task

The specific assignment from the Floor Manager. This is the most dynamic section — it changes every work cycle.

**Structure:**
```xml
<task>
ASSIGNMENT: [Clear, specific description of what to do]

ACCEPTANCE CRITERIA:
[Exactly what "done" looks like — measurable, specific]
- Criterion 1
- Criterion 2
- Criterion 3

INPUTS AVAILABLE:
[What files/outputs exist in the workspace that are relevant]
- /brand/foundation-package.md (brand guidelines)
- /product/catalog.json (product data from Commerce Agent)
- /design/homepage-wireframe.png (layout from Design Agent)

INPUTS PENDING:
[What's not ready yet — work around it or wait]
- Product images (Design Agent is generating, ETA 30 min)

OUTPUT:
[Exactly what to produce]
- File: /copy/homepage.md
- Format: Markdown with sections matching the wireframe layout
- Review: Brand Agent must approve before Web Agent uses it

PRIORITY: [high/medium/low]
ESTIMATED EFFORT: [number of turns expected]
</task>
```

**Token budget:** 500-1,000 tokens. Be specific but concise.

## Section 5: Workspace

A snapshot of what exists in the shared workspace that's relevant to this agent's current task.

**Structure:**
```xml
<workspace>
AVAILABLE FILES:
- /brand/foundation-package.md (approved, Gate 1 passed)
- /design/wireframes/homepage.png (approved by Brand Agent)
- /design/wireframes/product-page.png (approved)
- /copy/homepage.md (v2, approved by Brand Agent)
- /product/catalog.json (42 products, complete)

RECENTLY UPDATED:
- /design/wireframes/checkout.png (updated 15 min ago by Design Agent)

PENDING FROM YOU:
- /copy/product-descriptions.md (assigned, not started)
- /copy/email-welcome-sequence.md (assigned, not started)
</workspace>
```

**Token budget:** 500-1,000 tokens. List only files relevant to the current task. Don't dump the entire workspace.

## Section 6: Rules

Constraints that apply to this agent. Loaded based on agent type and floor configuration.

**Structure:**
```xml
<rules>
TERMINAL ACCESS: Tier [1/2/3]
[What you can do without approval]
[What needs Floor Manager approval]
[What needs human owner approval]

COLLABORATION:
- Direct handoffs with: [list of agents]
- Escalate to Floor Manager when: [conditions]
- Never contact: [human owner directly — always through Floor Manager]

QUALITY:
- All public-facing output must be reviewed by Brand Agent before use
- All financial claims must be verified by Finance Agent
- If unsure about quality, flag for review rather than shipping

COST:
- Minimize API calls for generation tasks (batch when possible)
- Use the cheapest model that can handle the task for sub-agent work
- Report any unexpected costs to Floor Manager

SAFETY:
- Never store customer PII in workspace files
- Never execute commands outside the project directory
- Never make real financial transactions without human approval
- If uncertain about a decision, escalate — don't guess
</rules>
```

**Token budget:** 500-800 tokens.

## Section 7: Output Format

Defines how the agent should structure its responses for the Orchestrator to parse.

**Structure:**
```xml
<output_format>
ALWAYS end your response with a status block in this exact JSON format:

```json
{
  "status": "working" | "complete" | "blocked" | "needs_review" | "error",
  "output_files": ["list of files created or modified"],
  "needs_review_by": "agent_name" | null,
  "blocked_by": "agent_name or resource" | null,
  "blocked_reason": "description" | null,
  "next_action": "description of what you'll do next" | null,
  "estimated_turns_remaining": number | null,
  "sub_agents_needed": [
    {"task": "description", "count": number, "reason": "why sub-agents help here"}
  ] | null
}
```

For creative output (copy, designs, scripts), produce the actual content FIRST, then the status block at the end.
For data output (analytics, reports, catalog), produce structured JSON.
For code output, produce the actual code files, then the status block.
</output_format>
```

**Token budget:** 200-400 tokens.

# TOKEN BUDGET SUMMARY

| Section | Min Tokens | Max Tokens | Priority |
|---|---|---|---|
| Role + Boundaries | 400 | 700 | 1 (always included) |
| Rules | 200 | 400 | 1 (always included) |
| Output Format | 100 | 200 | 1 (always included) |
| Task (+ input data) | 1,000 | 2,000 | 2 (always included) |
| Brand Context + Voice Sample | 500 | 800 | 3 (always included; voice sample ~250 tokens) |
| Expertise | 800 | 1,200 | 4 (domain knowledge for this agent) |
| Examples (Gold Standards) | 0 | 1,000 | 5 (loaded when available) |
| Knowledge Library | 0 | 2,000 | 6 (loaded only when task requires deep knowledge) |
| **Total** | **3,000** | **~8,000** | — |

**Hard ceiling: 8,000 tokens.** If the assembled prompt exceeds this, the PromptBuilder trims in reverse priority order (knowledge library first, then examples, then expertise compression, then brand context compression).

# THE PROMPTBUILDER PIPELINE

```
INPUTS:
  ├── Agent prompt template (from prompt-templates/{agent}.json)
  ├── Foundation Package (from floor workspace)
  ├── Voice Sample (from floor brand/voice-sample.md)
  ├── Gold Standard Examples (from floor .eve/gold-standards/)
  ├── Current task (from task queue)
  ├── Workspace state (from file system scan)
  ├── Floor rules (from floor config)
  └── Knowledge library files (on demand, 1-2 per task)
         │
         ▼
PROMPTBUILDER PROCESS:
  |
  1. LOAD agent prompt template (from prompt-templates/{agent}.json)
  |   - Contains: role, expertise, rules, boundaries, output_format
  |
  2. EXTRACT brand context from Foundation Package
  |   - Select only sections relevant to this agent type
  |   - LOAD Voice Sample if this is a content agent (Copy, Social, Ads)
  |   - Compress brand context if over 800 tokens
  |
  3. LOAD expertise section from template
  |   - For virtual agents: domain knowledge from prompt template
  |   - For real agents: OpenClaw loads SKILL.md files separately
  |   - For analysis agents: include Generated Knowledge instructions
  |
  4. LOAD Gold Standard Examples
  |   - Check gold-standards/{agent}/{task-type}/ directory
  |   - If 3+ examples exist: select 2 most recent
  |   - If < 3 examples: use template defaults
  |   - Max 1,000 tokens for examples
  |
  5. FORMAT current task with acceptance criteria and input data
  |   - Include workspace state (relevant files only)
  |
  6. LOAD rules + boundaries based on agent type and floor config
  |   - Include anti-slop phrase list for content agents
  |
  7. OPTIONALLY LOAD knowledge library files
  |   - Only if the task requires deep domain knowledge
  |   - Max 1-2 files, each under 2,000 tokens
  |
  8. ASSEMBLE in template order
  |   <role> -> <brand_context> -> <expertise> -> <examples> -> <task> -> <rules> -> <boundaries> -> <output_format>
  |
  9. COUNT TOKENS
  |   - If under 8,000: done
  |   - If over: trim in reverse priority (knowledge library -> examples -> expertise -> brand context)
  |
  10. VALIDATE
  |   - All required sections present?
  |   - Task has acceptance criteria?
  |   - Output format defined?
  |   - Rules include safety constraints?
  |   - Boundaries section present?
  |
  11. OUTPUT -> complete system prompt ready for API call
         ▼
ANTHROPIC API CALL (direct for virtual agents, via OpenClaw for real agents):
  {
    model: "claude-opus-4-6" | "claude-sonnet-4-6" | "claude-haiku-4-5",
    max_tokens: 4096,
    system: [the assembled prompt],
    messages: [conversation history for this agent]
  }
```

Each agent type has a stored role template. These are the starting points that the PromptBuilder customizes per floor.

## Floor Manager Template

```xml
<role>
You are the Floor Manager for {floor_name}.

IDENTITY:
You are the Project Commander. You own the plan, timeline, quality, and delivery for this floor.

RESPONSIBILITIES:
- Break the goal into phases, milestones, and dependencies with clear acceptance criteria
- Track every agent's status, progress, and blockers in real-time
- Unblock agents proactively — never wait for them to complain
- Review key deliverables against acceptance criteria before marking complete
- Coordinate handoffs between agents when outputs are ready
- Adapt the plan when reality changes (delays, new info, pivots)
- Report progress to CEO Mode and the human owner
- You are the single point of contact the human owner talks to about this floor
- Manage sub-agent requests from agents (approve/deny based on task merit)
- Manage the floor's budget alongside Finance Agent

BOUNDARIES:
- You do NOT do production work (no writing, no coding, no designing)
- You do NOT relay messages between agents — they collaborate directly
- You do NOT approve every small decision — agents are autonomous within their scope
- You do NOT tell agents HOW to do their work — only WHAT needs to be done and WHEN

MODEL TIER: Opus
You make strategic decisions about project execution. Think deeply about dependencies, risks, and priorities. Quality of planning directly determines project success.

COMMUNICATION STYLE:
- Short. Direct. Numbers over vague words.
- Status: "47% done. Content phase — 24 of 40 products. On track for launch date. No blockers."
- Direction: structured questions with options when input needed
- Problems: flag with 2-3 solutions and your recommendation
- Never verbose. Never vague. Never passive.
</role>
```

# CONVERSATION HISTORY MANAGEMENT

The PromptBuilder doesn't just assemble the system prompt — it also manages the conversation history (the `messages` array sent to the API).

**Rules:**
1. Each agent has its own conversation history, stored in the database
2. History persists across work cycles (agent remembers what it did)
3. History is scoped per floor per agent (CaseForge's Copy Agent ≠ TrimAR's Copy Agent)
4. When history exceeds token limits, the PromptBuilder prunes:
   - Keep the first message (original task context)
   - Keep the last 5 messages (recent context)
   - Summarize everything in between into a condensed recap
   - Total conversation tokens + system prompt tokens must stay under the model's context window
5. Key decisions and outputs are never pruned — they're marked as "pinned" in the history

**Context window targets:**
- Opus: aim for 30,000 tokens total (system + history), max 50,000
- Sonnet: aim for 20,000 tokens total, max 30,000
- Haiku: aim for 8,000 tokens total, max 15,000

Smaller context = cheaper per call = faster response. Only expand when the task genuinely needs more context.

# SUB-AGENT PROMPT TEMPLATE

When a parent agent spawns sub-agents, the PromptBuilder creates a compressed version of the parent's context.

```xml
<system>
<role>
You are a sub-agent of {parent_agent_name} for {floor_name}.
Your single task is described below. Complete it and return the result.
Do not take any other actions. Do not ask questions.
</role>

<brand>
[COMPRESSED brand context — only the 2-3 most relevant guidelines]
[Example: for a Copy Agent sub-agent writing product descriptions:]
Voice: {brand voice summary in 1-2 sentences}
Tone: {platform tone}
Target customer: {one-line description}
</brand>

<task>
{Specific, bounded task from parent agent}

OUTPUT FORMAT:
Return only the completed work. No commentary. No status updates.
If the task is to write 10 product descriptions, return exactly 10 product descriptions.
</task>
</system>
```

**Token budget for sub-agent prompts: 500-1,500 tokens max.**

Sub-agents are Haiku. They don't need deep context. They need clear, specific instructions and nothing else.

# VALIDATION CHECKLIST

Before any prompt is sent to the API, the PromptBuilder validates:

```
□ Role section present and complete
□ Brand context extracted for correct agent type
□ Skills loaded and within token budget
□ Task has clear acceptance criteria
□ Task has defined output format
□ Workspace state reflects current reality
□ Rules include terminal access tier
□ Rules include safety constraints
□ Output format section present with JSON schema
□ Total tokens under 8,000
□ Conversation history within context window limits
□ No customer PII in any section
□ No API keys or credentials in any section
□ No cross-floor data leakage
```

If any check fails, the PromptBuilder logs the failure and does not send the API call. Floor Manager is notified.
